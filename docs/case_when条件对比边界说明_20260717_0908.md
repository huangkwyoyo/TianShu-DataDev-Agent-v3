# CASE WHEN 条件对比边界说明

> 记录 SQL ↔ Spark 逻辑对比中 `case_when` 步骤判定为 UNSUPPORTED 的设计取舍。

## 问题背景

通过 **Systematic Debugging** 对 Pipeline 全流程排查发现，`PlanComparator` 输出的逻辑对比报告中，CASE WHEN 步骤的判定结果为 `UNSUPPORTED_COMPARISON`，而 filter、project、scan 等均已达到 EQUIVALENT。

追踪数据流：

1. Builder 为 label_table 生成 `CaseWhenStep`，每个 `WhenBranch` 携带结构化条件（如 `distance_miles <= 2`）
2. `_flatten_case_when_step()` 检测到 condition 非空，设置 `has_conditions = True`
3. `compare_case_when_steps()` 发现 `has_conditions = True`，按设计主动返回 UNSUPPORTED

## 当前覆盖情况

### ✅ 已等价验证（结构骨架）

| 字段 | 说明 |
|------|------|
| labels | 所有 WHEN 分支的 result.value 值（如 unknown/short/medium/long） |
| else_value / default_value | ELSE 子句的默认值 |
| output_alias | CASE WHEN 结果的输出列别名 |

### ❌ 未覆盖（业务语义核心）

| 字段 | 说明 |
|------|------|
| **condition** | 每个 WHEN 分支的谓词条件（如 `distance_miles <= 2`）—— 真正决定数据落入哪个标签的逻辑 |

## 设计取舍

### 方案 A：实现 condition 语义等价对比（否决）

复用 `compare_filter_steps` 的 Predicate 递归对比逻辑。否决理由：

- condition 中可能含 ColumnRef 表别名，需额外归一化
- 语义等价但写法不同的谓词（如 `a OR (b AND c)` vs `(a OR b) AND (a OR c)`）会导致误判
- 对个人使用且强调安全边界的项目是过度建设

### ✅ 方案 B：保持现状，不修改代码（当前选择）

`compare_case_when_steps` 保留显式阻断，不实现 condition 对比。

## 准确表述约定

| 概念 | 准确表述 | 禁用表述 |
|------|----------|----------|
| labels/else_value/alias | 结构骨架 | "已覆盖业务语义" |
| condition | 未覆盖，待人工审核 | "逻辑等价" |
| 物理验证一致 | **CONSISTENT_SAMPLE**（当前快照一致） | "业务正确"、"逻辑等价" |

## 判定层级

```
静态层（PlanComparator）
  └─ case_when → UNSUPPORTED_COMPARISON
      └─ 结构骨架通过（labels/else_value/alias）
      └─ condition 待人工审核

动态层（Physical Verifier / Snapshot）
  └─ 样本一致 → CONSISTENT_SAMPLE
  └─ 样本不一致 → 数据差异
  └─ 不能从样本推全量，不能声称"逻辑等价"

最终判断：人工审核 condition 文本，确认业务逻辑正确
```

## 后续方向

- 此边界已记录，无需立即处理
- 仅当真实使用中频繁受阻、且积累足够多的 condition 等价样例后再考虑建设
- 届时可从 `compare_filter_steps` 复用 Predicate 递归对比逻辑
