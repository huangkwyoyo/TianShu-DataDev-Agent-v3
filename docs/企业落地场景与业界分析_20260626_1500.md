### 1.2 场景详解

#### A1：日常宽表/汇总表开发（最高频场景）

**现状痛点**：
- 数据工程师每天需要写大量聚合 SQL（`GROUP BY` + `SUM/COUNT/DISTINCT` + `ROW_NUMBER`）
- 需求文档与 SQL 代码之间靠人工翻译，易出错
- Join 逻辑分散在多个工程师脑中，缺乏系统化记录
- 代码审查时无法快速判断 SQL 是否完整覆盖了业务需求

**本项目如何解决**：
- 开发者写出 DeveloperSpec（Markdown + YAML 元数据），系统自动生成类型化 SqlBuildPlan → 确定性编译 SQL
- Join 推理由 LLM 提出候选 + Validator 证据定级 + 人工确认，Join 逻辑可追溯
- 产物附带 Code Review Package，审查者可对照需求逐项验证

**量化价值**（预估）：
- 单表聚合类需求：开发时间从 30min → 10min（提效 67%）
- Join 汇总类需求：开发时间从 2h → 30min（提效 75%）
- 代码审查时间：从 30min → 10min（提效 67%，因审查材料结构化）

---

#### A2：多步骤 ETL 加工（高价值场景）

**现状痛点**：
- 多步骤 ETL 涉及中间表、临时表、依赖排序
- 中间表字段追踪困难，改了上游不知道下游影响
- 每步结果验证靠肉眼 `SELECT COUNT(*)`，缺乏系统化

**本项目如何解决**：
- SqlProgram（多语句 DAG）自动编排步骤依赖和拓扑排序
- 每步中间表行数预估与实际执行对比，差异 >20% 标 WARN
- 全链路字段血缘可追溯（SourceManifest → SqlBuildPlan → SqlProgram）

**量化价值**（预估）：
- 3-5 步 ETL 开发：从 1 天 → 2h（提效 75%）
- 中间步骤调试：从反复执行验证 → 自动行数校验 + 差异报告

---

#### A4：SQL→PySpark 迁移与验证（平台级场景）

**现状痛点**：
- 企业从数据仓库向数据湖/Spark 迁移时，需要人工重写所有 SQL 为 PySpark
- 两套实现的等价性验证极度困难——通常是"跑出来的数差不多就行"
- 迁移周期长（数月到数年），风险高

**本项目如何解决（Phase 5-8）**：
- 从已验证的 SqlBuildPlan 确定性抽取 DataTransformContract
- SparkDeveloper 在契约约束下生成受控 PySpark 纯转换函数
- 双链验证：PlanEquivalence（逻辑结构等价）+ ResultComparator（物理执行结果一致）

**量化价值**（预估）：
- 单张表迁移：从 3-5 天 → 1 天（提效 70-80%）
- 等价性验证：从"靠人眼对比" → "确定性 Comparator 给出 CONSISTENT_SAMPLE"
- 迁移项目周期：从 6-12 个月 → 3-6 个月

---

#### C1：跨引擎一致性验证——审计/合规场景（高壁垒场景）

**现状痛点**：
- 金融、保险、证券等强监管行业，关键数据加工逻辑需要双重验证
- 监管要求"同一业务逻辑在不同引擎上的结果必须一致"
- 目前几乎全靠人工抽查 + 文档承诺，缺乏自动化手段

**本项目如何解决**：
- SQL + PySpark 双引擎在同一个不可变快照上真实执行
- 确定性 Comparator 给出 CONSISTENT_SAMPLE（不是 "PASS"）
- 完整的 provenance.yml 审计链：代码哈希、Prompt 版本、模型标识、快照哈希

**这是本项目区别于所有竞品的核心壁垒**——目前业界没有任何工具能在 SQL 和 PySpark 之间做确定性的等价性交叉验证。

### 2.3 本项目的差异化壁垒

```
┌─────────────────────────────────────────────────────────────┐
│                    差异化核心壁垒                              │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  1. 确定性验证，而非 LLM 裁决                                  │
│     ├── Validator 是 Python 确定性代码，不是 Prompt            │
│     ├── Comparator 是算法比对，不是 LLM "你觉得一致吗"          │
│     └── PerfValidator 是规则引擎，不是 LLM 建议               │
│                                                             │
│  2. 类型化中间表示（IR），而非自由 SQL 文本                      │
│     ├── 8 种封闭 Step 类型（Scan/Filter/Join/Aggregate/...)    │
│     ├── 禁止 raw_sql、where_sql、join_on: str                 │
│     └── Python 确定性编译器 → SQL（非 LLM 生成 SQL）            │
│                                                             │
│  3. 双链交叉验证（SQL ↔ Spark）                                │
│     ├── PlanEquivalence：逻辑结构等价比较                       │
│     ├── ResultComparator：物理执行结果一致性                     │
│     └── 同一不可变快照，禁止分别抽样                             │
│                                                             │
│  4. 受控代码生成（PySpark）                                    │
│     ├── 固定函数签名 transform(inputs, params) → DataFrame     │
│     ├── 禁止 Action、Sink、UDF、网络、文件系统                    │
│     └── Static Validator 编译期拦截违规                         │
│                                                             │
│  5. 完整审计链（Provenance）                                   │
│     ├── 每次生成记录：代码哈希 + Prompt 版本 + 模型标识           │
│     ├── 快照哈希 + 执行时间戳                                  │
│     └── 满足金融/保险行业合规审计要求                             │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

**一句话定位**：在 Text2SQL（太浅）和全自动数据平台（太泛）之间，本项目占据"AI 辅助数据开发工作台"这一空白地带——**不是帮你写 SQL，而是帮你交付可审查的工程代码**。

### 3.5 Token 消耗优化策略

按本项目架构已内置的优化 + 可额外实施的策略：

| 优化策略 | 节省比例 | 实现阶段 | 说明 |
|----------|---------|---------|------|
| **Prompt Caching**（系统 Prompt） | 15–25% | Phase 4A | Anthropic 原生支持，系统 Prompt 缓存 5 分钟 TTL |
| **结构化输出缓存**（相同输入→相同哈希→返回缓存） | 30–50% | Phase 4B | SqlBuildPlan 哈希作为缓存 Key，相同 DeveloperSpec 不重复生成 |
| **模型分层调用**（关键步骤 Opus，辅助步骤 Haiku） | 40–60% | Phase 4A | Join 推理用 Sonnet，审查包草稿用 Haiku，关键计划用 Opus |
| **减少 Repair 返工**（提升首轮准确率） | 20–40% | Phase 4+ | Harness 持续优化 Prompt，目标首轮通过率 ≥ 80% |
| **批量处理**（相近 DeveloperSpec 共享上下文） | 10–20% | Phase 8 | 同项目多个相关需求共享 Schema 上下文，减少重复输入 |
| **Memory 知识库**（复用历史 Join 推理） | 5–15% | Phase 7 | 常见 Join 模式直接查询 Memory，跳过 LLM 推理 |

**综合优化后**（缓存 + 分层 + Memory），单次双链平均成本预估：

| 模型组合 | 优化前 | 优化后 | 降幅 |
|----------|--------|--------|------|
| Opus × 6 次调用 | $1.20 | $0.35–$0.50 | 60–70% |
| Sonnet × 6 次调用 | $0.24 | $0.07–$0.10 | 60–70% |
| 混合（Haiku + Sonnet + Opus） | — | $0.15–$0.25 | — |

## 四、总结

### 企业落地的核心判断

```
适合落地的企业特征：
├── 有数据仓库/数据湖开发团队（≥10 人）
├── 日常 ETL/宽表开发需求密集（≥50 张表/月）
├── 有代码审查流程（或想建立）
├── 数据合规/审计需求（金融、保险、证券加分）
└── 正在进行或计划 SQL→Spark 迁移

不适合的场景：
├── 业务人员直接问数（不是本产品定位）
├── 数据开发需求零散（<10 张表/月，ROI 不够）
├── 已有成熟的低代码 ETL 平台（如 DataWorks + AI 已够用）
└── 不接受 LLM 参与开发流程的组织
```

### Token 成本的底线判断

- 日常 SQL 开发（Sonnet 级）：**$0.12–0.21/条需求**，低于数据工程师 2 分钟的人力成本
- 完整双链验证（Sonnet 级）：**$0.24–0.43/条需求**，仍低于 5 分钟的人力成本
- 综合优化后可再降 60–70%，单位成本进入"几分钱"区间
- **Token 成本不是瓶颈**——真正的瓶颈是 LLM 结构化输出的稳定性和双链验证的覆盖度

### 一句话结论

> 本项目在"数据开发工程化"这个蓝海市场中占据了一个独特的定位——不做 Text2SQL 的红海竞争，不做通用 AI 编程的边缘功能，而是通过**确定性验证 + 双链交叉验证 + 受控代码生成**构建了一套从需求到可审查代码的完整工程化流水线。在金融合规和跨引擎迁移这两个高价值场景中，目前没有直接竞品。
