# CASE WHEN condition UNSUPPORTED 物理验证白名单设计（最终精简版）

> **分类**：C 类（架构风险）
> **状态**：设计阶段——等待 Owner 批准后实施
> **日期**：2026-07-17（修订 v3 — 最终精简）

---

## 1. 背景

当前 `compare_case_when_steps` 检测到 condition（谓词条件）时返回 `UNSUPPORTED_COMPARISON`，Pipeline 门禁跳过物理验证。condition 不可自动判定等价时，双引擎物理执行仍然可行——行数和样本数据可为人工审核提供交叉证据。

白名单场景：**Validator 通过 + comparator_report 中只有 `case_when` 因 condition 为 `UNSUPPORTED_COMPARISON` + 其他 step 全部 `EQUIVALENT`** 时放行物理验证，结果作为人审证据（不改变 `HUMAN_REVIEW_REQUIRED` 状态）。

## 2. 风险分析

### 2.1 当前状态

| 入口 | Validator 门禁 | Comparator 门禁 | 问题 |
|------|:--------------:|:---------------:|------|
| `run_all_full`（同步） | ✅ | **无** | PHYSICAL_VERIFIER 在 Validator 通过后总被执行 |
| `run_all_full_stream`（SSE） | ✅ | ✅ 非 `LOGIC_EQUIVALENT` → 跳过 | case_when UNSUPPORTED 被跳过 |
| `run_spark_stage`（独立） | **无**（仅依赖检查） | **无** | 可绕过门禁 |

### 2.2 约束

- 不新增验证 API、状态枚举、前端组件
- 物理结果仅为人审证据——即使一致也**不得**进入 REVIEW_READY
- 窄白名单：仅 case_when-condition UNSUPPORTED + 其他 step 全部 EQUIVALENT
- 非白名单保持跳过

### 2.3 影响

- **安全**：无放宽——物理验证只读，不变更状态流转
- **性能**：白名单场景触发 Spark 执行（+30-60s），但限于极窄窗口
- **状态**：`HUMAN_REVIEW_REQUIRED` 不变

## 3. v2.0 边界检查

| 边界 | 判定 |
|------|:----:|
| Validator → Executor | ✅ 物理验证在 Executor 层，Validator 必须先通过 |
| 防线 2 | ✅ 不修改人审闸门、REVIEW_READY 判定 |
| 交叉验证 | ✅ 物理验证是交叉验证的组成部分 |
| 来源追溯 | ✅ 不改变 artifact hash 链 |

## 4. 设计方案

### 4.1 SparkStageContext 新增字段

在 `SparkStageContext` 新增 `physical_verify_report`，保存完整的物理验证报告（`PhysicalVerificationReport`），供下游排查诊断：

```python
@dataclass
class SparkStageContext:
    # ... 现有字段 ...
    comparator_report: PlanComparisonReport | None = None
    # ── 新增 ──
    physical_verify_report: PhysicalVerificationReport | None = None
```

**`run_spark_stage` 返回有界摘要**：

```python
# PHYSICAL_VERIFIER 成功时
result = {
    "type": "physical_verify",
    "status": "ok" | "failed",       # 物理一致/不一致
    "row_count_match": True | False,
    "schema_match": True | False,
    "total_diff_count": 0,            # 差异行总数
    "sample_rows": {                  # 两侧最多 5 条
        "duckdb": [{"col": "val"}, ...],   # duckdb_result.sample_rows[:5]
        "spark": [{"col": "val"}, ...],    # spark_result.sample_rows[:5]
    },
    "skipped": False,
}

# PHYSICAL_VERIFIER 跳过时
result = {
    "type": "physical_verify",
    "skipped": True,
    "message": "...",
}
```

> `PhysicalVerificationReport.duckdb_result.sample_rows` 和 `.spark_result.sample_rows` 在现有代码中已有赋值（行 651/684/800/884 等），`run_spark_stage` 仅需从中提取前 5 条。

### 4.2 统一门禁——仅在 `_do_spark_physical_verify` 一个位置

`_should_physical_verify` 返回 **bool**（不预留字符串模式），放在 `_do_spark_physical_verify` 入口处：

```python
@staticmethod
def _should_physical_verify(
    validator_ok: bool,
    comparator_report: PlanComparisonReport | None,
) -> bool:
    """判定 PHYSICAL_VERIFIER 是否应执行。"""
    if not validator_ok:
        return False          # Validator 未通过 → 跳过

    if comparator_report is None:
        return False          # 无对比报告 → 跳过

    # 条件 A：LOGIC_EQUIVALENT → 正常执行
    if comparator_report.status == ComparisonStatus.LOGIC_EQUIVALENT:
        return True

    # 条件 B：白名单——仅 case_when 因 condition 为 UNSUPPORTED
    if comparator_report.status == ComparisonStatus.LOGIC_UNSUPPORTED:
        has_case_when_unsupported = False
        for r in comparator_report.step_results:
            if r.verdict == EquivalenceVerdict.NOT_EQUIVALENT:
                return False          # 任何不等价 → 跳过
            if r.verdict == EquivalenceVerdict.UNSUPPORTED_COMPARISON:
                if r.step_type != "case_when":
                    return False      # 非 case_when → 跳过
                has_case_when_unsupported = True
        return has_case_when_unsupported

    # 其他情况 → 跳过
    return False
```

**`_do_spark_physical_verify` 内新增门禁检查**（在 Step 1 之前）：

```python
def _do_spark_physical_verify(self, artifacts, context):
    # ── 门禁检查（新增） ──
    validator_ok = context.stage_results.get("VALIDATOR") == "SUCCESS"
    if not self._should_physical_verify(validator_ok, context.comparator_report):
        context.stage_results["PHYSICAL_VERIFIER"] = "SKIPPED"
        context.errors.append(
            "[PHYSICAL_VERIFIER] SKIPPED: 物理验证门禁未通过"
            "（Validator 未通过，或 Comparator 不满足白名单条件）"
        )
        return

    # Step 1：检查 PySpark 运行时环境（现有逻辑）
    try:
        import pyspark
    except ImportError:
        ...

    # ... 后续执行逻辑不变 ...
    
    # ── 保存完整报告到 SparkStageContext（新增） ──
    # 在 verify() 完成后：
    context.physical_verify_report = report
```

**物理验证成功后**，`run_spark_stage` 从 `context.physical_verify_report` 中提取有界摘要返回。

### 4.3 三条入口的行为变化

**`run_all_full`（同步）**：
- 不再需要独立的 PHYSICAL_VERIFIER Validator 门禁（行 2228-2237，已简化）
- `run_spark_stage` 内部的 `_do_spark_physical_verify` 处理门禁
- `spark_ok` 安全网独立于门禁

**`run_all_full_stream`（SSE）**：
- 删除独立的 COMPARATOR 门禁（行 2488-2501）——`_do_spark_physical_verify` 统一处理
- SSE 事件分发逻辑保持不变，只去掉重复门禁

**`run_spark_stage`（独立）**：
- 行为不变——`run_spark_stage` 已调用 `_do_spark_physical_verify`，门禁在入口处生效

### 4.4 状态安全网（三处独立不变）

| 位置 | 规则 | 变更 |
|------|------|:----:|
| `run_all_full` 汇总（行 2254） | `spark_ok = physver_stage exists AND status=="ok" AND comparator_status=="LOGIC_EQUIVALENT"` | 新增 `comparator_status` 条件 |
| `derive_overall_status`（orchestrator.py:156） | `LOGIC_UNSUPPORTED` → `HUMAN_REVIEW_REQUIRED` | 不变 |
| `_compute_review_ready`（review_builder.py:185） | `comparator_status not in ("LOGIC_EQUIVALENT",)` → `review_ready=False` | 不变 |

在 `run_all_full` 和 `run_all_full_stream` 的响应中**显式返回**：

```python
# 汇总响应增加以下字段：
{
    "comparator_status": ...,          # 已有的细粒度状态
    "requires_human_review": True,     # 非 LOGIC_EQUIVALENT 时固定 True
    "review_ready": False,             # 非 LOGIC_EQUIVALENT 时固定 False
}
```

白名单场景始终：`spark_ok=False`、`requires_human_review=True`、`review_ready=False`。

### 4.5 前端处理

- **样本不通过前端直接展示**——`sample_rows` 仅通过 API 响应返回（`run_spark_stage` 和 `run_all_full` 的 JSON 响应）
- 若后续需要页面查看，**只扩展现有物理验证面板**，不新增组件

### 4.6 物理不一致的状态处理

当双引擎结果不一致时：

```python
# PhysicalVerifier 返回 RESULT_MISMATCH → 映射为
context.stage_results["PHYSICAL_VERIFIER"] = "SUCCESS"  # 验证执行成功
# 但 run_spark_stage 响应中 status → "failed"
# 有界报告保留在 context.physical_verify_report 中供诊断
```

映射规则：
| 物理验证状态 | stage_results | run_spark_stage status |
|:------------:|:-------------:|:----------------------:|
| `RESULT_CONSISTENT` | `SUCCESS` | `"ok"` |
| `RESULT_MISMATCH` | `SUCCESS` | `"failed"` |
| 白名单跳过 | `SKIPPED` | `"skipped"` |
| PySpark 不可用/缺产物 | `SKIPPED` / `FAILURE` | `"skipped"` / `"failed"` |

**物理不一致时**不触发整体管线失败——仅标记该阶段为 `failed`，其他阶段状态不受影响。`context.physical_verify_report` 保留完整差异报告供 API 查询和诊断。

## 5. 替代方案

| 方案 | 优点 | 缺点 |
|------|------|------|
| **★ 推荐：单门禁在 _do_spark_physical_verify** | 改动最少（1 处核心逻辑 + 1 个静态方法） | 无 |
| 方案 B：三入口各加门禁 | 门禁显式 | 三处重复，后续改动容易漏 |
| 方案 C：新增 lightweight verify | 语义清晰 | 违背"不新增 API/方法"约束 |

## 8. 架构边界影响

| 维度 | 判定 |
|------|:----:|
| 是否改变双层边界（Validator→Executor） | **否**——物理验证在 Executor 层，Validator 必须通过 |
| 是否破坏防线 2 | **否**——REVIEW_READY 判定不变，人审闸门不变 |
| 是否绕过 Validator→Executor | **否**——Validator 必须先通过 |
| 是否放宽安全限制 | **否**——物理验证只读 |
| 是否授予生产库权限 | **否**——不涉及 |

**结论**：不改架构边界，不改状态流转，不改 REVIEW_READY 判定。物理结果严格限定为"人工审核证据"。
