# CRE v2——Canonical Row Encoding 双引擎确定性比较设计

> 设计版本：v2.3（最终准入硬化完成）
> 设计日期：2026-07-13
> 状态：shadow 模式已完成最终准入硬化（EnvironmentManifest 唯一权威、Any/dict 逃生口清除、Artifact Store 集成、Singleton 统一 DecisionEngine、零容忍准入标准），等待 Owner 批准正式接管生产结论（Phase 3）

---

## 1. 背景与动机

当前 `PhysicalVerifier.verify()` 的全量比较方式是逐行将 DuckDB 和 Spark 的结果拉取到内存后进行逐字段比较。这种方式存在以下问题：

1. **数据传输开销大**：结果集全量通过网络传输到 Python 进程，行数越多延迟越高
2. **比较粒度粗**：缺少诊断定位能力——只知道"不一致"，不知道"哪些行、哪些列"
3. **无可容忍差异机制**：浮点尾差的微小差异（1e-15 级别）直接判为 MISMATCH，实际是引擎间 IEEE 754 运算顺序差异
4. **缺少架构隔离**：比较逻辑与验证流程耦合在同一个 `verify()` 方法中

**优化方向**："全量行数/schema + 全量确定性哈希 + 分桶诊断定位 + 主键样本展示"——在不削弱验证边界的前提下显著降低数据传输和内存占用。

## 2. CRE 编码协议

### 2.1 整体格式

```
[4B ]  magic          "CRE2" (0x43524532)
[2B ]  column_count   uint16 BE
[4B ]  total_length   uint32 BE（含本字段之前的所有字节）
[1B ]  type_tag_1
[N1B]  value_bytes_1
[1B ]  type_tag_2
[N2B]  value_bytes_2
...
```

### 2.2 类型标签

| 标签 | 类型 | 值编码 | 长度 |
|------|------|--------|------|
| 0x00 | NULL | 无值字节 | 1 |
| 0x01 | BOOLEAN | `0x00`(false) / `0x01`(true) | 2 |
| 0x02 | INT8 | 补码 1 字节 | 2 |
| 0x03 | INT16 | Big-endian 补码 2 字节 | 3 |
| 0x04 | INT32 | Big-endian 补码 4 字节 | 5 |
| 0x05 | INT64 | Big-endian 补码 8 字节 | 9 |
| 0x06 | FLOAT32 | IEEE 754 binary32, big-endian | 5 |
| 0x07 | FLOAT64 | IEEE 754 binary64, big-endian | 9 |
| 0x08 | DECIMAL | `[1B]0x08 [1B]precision [1B]scale [1B]unscaled_byte_length [N B]unscaled_value` | 变长 |
| 0x09 | VARCHAR | 长度(4B) + UTF-8 字节 | 变长 |
| 0x0A | DATE | 距 1970-01-01 天数(int32 BE) | 5 |
| 0x0B | TIMESTAMP | 距 1970-01-01 微秒数(int64 BE) | 9 |

### 2.3 编码规则

- **NULL**：所有类型的 NULL 统一编码为单字节 `0x00`（无值字节）
- **BOOLEAN**：`False`→`0x00`，`True`→`0x01`。int 仅接受 0(False)/1(True)，其他整数拒绝。字符串白名单：`true/false/1/0/yes/no/t/f`（大小写不敏感）
- **整数**：统一走补码 big-endian，长度由类型精度决定（8→1B, 16→2B, 32→4B, 64→8B）
- **FLOAT/DOUBLE**：使用 `struct.pack(">f"/">d")` 保持原生 IEEE 754 编码，**禁止** round、转 varchar、网格哈希等非可逆转换
- **DECIMAL**：按 `decimal(p,s)` 的 scale 量化（`ROUND_HALF_UP`）后编码 unscaled 整数值，比 `Decimal.to_eng_string()` 更紧凑。编码格式：`[1B]0x08 [1B]precision [1B]scale [1B]unscaled_byte_length [N B]unscaled_value`（big-endian signed）。length prefix 确保解码器知悉精准字节边界，与尾随零无关
- **VARCHAR**：`str(value).encode("utf-8")` + 4 字节长度前缀，使用二进制 Unicode 编码（无 NFC/NFD 归一化）
- **DATE**：`date.fromisoformat()` 解析后计算 epoch days，存 int32 BE
- **TIMESTAMP**：`datetime.fromisoformat()` 解析后转 UTC，存微秒级 int64 BE（需配置 timezone）

### 2.4 禁止规则

- **未知类型**（`_type_family()` 返回 `"UNKNOWN"`）：抛出 `ValueError`，禁止 fallback 到 `str(value)`
- **COMPLEX 类型**（array/struct/map）：抛出 `ValueError`
- **TIMESTAMP 无 timezone**：禁止编码，抛出 `ValueError`（强制配置时区避免歧义）

## 4. 容差规则

### 4.1 按类型族的比较策略

| 类型族 | 比较方法 | 配置参数 |
|--------|----------|----------|
| FLOAT / DOUBLE | `math.isclose(rel_tol, abs_tol)` | `float_rel_tolerance`, `float_abs_tolerance` |
| DECIMAL | `Decimal.quantize(Contract.scale)` 后精确 `==` | 从 `decimal(p,s)` 解析 scale |
| INT8~INT64 | 精确 `==`，无容差 | 无 |
| BOOLEAN | 精确 `==`，无容差 | 无 |
| VARCHAR | `str == str`，无容差 | 无 |
| DATE | epoch days `==`，无容差 | 无 |
| TIMESTAMP | UTC 微秒 `==`，无容差 | `CreConfig.timezone` |
| COMPLEX | 拒绝（抛出 ValueError）| 无 |
| UNKNOWN | 拒绝（抛出 ValueError）| 无 |

### 4.2 关键设计决策

- **float/double 禁止非可逆转换**：不 round、不转 varchar、不网格化哈希。仅用 `math.isclose` 判断是否在容差内，原始值之间的差异在 `CellComparisonResult.abs_error/rel_error` 中记录
- **Decimal 禁止转 float**：`Decimal(str(value))` 后按 Contract scale 量化，避免 float 精度损失导致误报
- **浮点差异的 WARN 模型**：所有容差内差异汇总为 `ToleratedDifferenceWarning`，包含受影响列、行数、比例、最大误差和最多 10 个样本快照

## 5. 判定矩阵

### 5.1 决策优先级（Gate 1→7）

```
Gate 1:  缺少 Config 配置（output_columns / primary_keys / timezone）
          → HUMAN_REVIEW

Gate 2:  对齐失败（error_message && !duplicate_keys）
          → HUMAN_REVIEW

Gate 3:  重复主键（duplicate_keys）
          → HUMAN_REVIEW（无法确定哪行是权威值）

Gate 4:  行数不匹配（duckdb_only || spark_only）
          → MISMATCH（引擎间产生了不同行数）

Gate 5:  不允许容差的类型差异（INT/BOOL/DATE/STRING/TIMESTAMP）
          → MISMATCH（这些类型必须精确一致）

Gate 6:  超容差的数值差异（OUT_OF_TOLERANCE）
          → MISMATCH（isclose 不满足）

Gate 7a: 未知原因 / 无规则
          → MISMATCH

Gate 7b: 缺少 data_type（data_type=None 但列存在）
          → HUMAN_REVIEW

通过:    全量 exact match
          → CONSISTENT

通过:    全量在容差内（within_count > 0）
          → CONSISTENT_WITH_WARN
```

### 5.2 关键约束

- **ratio 不改变判定**：即使 `tolerated_ratio > 5%`，只要所有差异都在已批准的容差内，结论仍是 `CONSISTENT_WITH_WARN`（不是 `HUMAN_REVIEW`）。ratio 仅影响 WARN 的严重度描述（"高"/"低"）
- **HUMAN_REVIEW 不等于 MISMATCH**：HUMAN_REVIEW 表示系统无法确定结论，需要人工判断。MISMATCH 表示系统确定不一致

## 6. 分桶诊断

### 6.1 分桶策略

- **桶数**：`CreConfig.num_buckets`（默认 64），推荐 2^n
- **分桶依据**：基于主键 digest（`SHA-256(pk_digest)`）的前 4 字节取模，不基于行 digest
- **桶内聚合**：桶内各行 的 row_digest（`SHA-256(encode_row())`）先排序后拼接，再计算桶级 digest

### 6.2 诊断价值

- 全桶 digest 一致 → 行级数据无差异（无需逐行比较）
- 某桶不一致 → 定位到该桶对应的主键范围缩小排查范围

## 7. 与生产 verify() 的关系

### 7.1 当前边界（Phase 2.5——shadow 集成）

- **影子模式**：`PhysicalVerifier.verify()` 在返回前并行调用 `_shadow_cre_diagnose()`，结果写入 `PhysicalVerificationReport.cre_shadow_report`（dict 字段）
- **不变约束**：shadow 不改变 legacy status、修复路由、REVIEW_READY 门禁；不修改 `executor.py` 输出协议
- **主键来源**：`primary_keys` 从 `DataTransformContract.grouping_keys` 提取，禁止 `primary_keys=[]` 硬编码；缺失时 shadow 标记 NOT_EXECUTED
- **状态映射**：CRE 原始状态通过 `_CRE_TO_LEGACY_MAP` 映射后与 legacy status 比较一致性

### 7.2 Shadow 架构

```
PhysicalVerifier.verify()
  │
  ├─ {... 生产验证流程：DuckDB + Spark 执行、规范化、diff 计算 ...}
  │
  └─ _shadow_cre_diagnose()
       │  (in: duckdb_raw, spark_raw, norm_config, legacy_status,
       │      primary_keys, timezone, environment_manifest)
       │
       ├─ 前置检查：缺少 output_columns/primary_keys → NOT_EXECUTED
       │
       ├─ 构造 CreConfig + EnvironmentManifest（从参数显式传入）
       │
       ├─ CREEncoder → KeyBasedRowAligner → ToleranceComparator → DecisionEngine
       │
       ├─ 状态映射：
       │    CONSISTENT           → RESULT_CONSISTENT
       │    CONSISTENT_WITH_WARN → RESULT_CONSISTENT
       │    MISMATCH             → RESULT_MISMATCH
       │    HUMAN_REVIEW         → HUMAN_REVIEW
       │    ERROR/其他           → NOT_EXECUTED
       │
       ├─ status_consistent = (mapped_status == legacy_status)
       ├─ 结论不同 → human_review_recommended=True
       │
       └─ 返回 dict（含：cre_status, mapped_status, legacy_status,
                      status_consistent, diagnostic_available,
                      human_review_recommended, decision_reason, warnings）
```

### 7.3 Shadow 报告结构

```python
{
    "cre_status": str,                     # CRE 原始状态
    "mapped_status": str,                  # 映射后的生产状态
    "legacy_status": str,                  # 生产 verify() 的原始状态
    "status_consistent": bool,             # mapped_status == legacy_status
    "diagnostic_available": bool,          # True=CRE 正常执行
    "human_review_recommended": bool,      # 两套结论不一致时提示
    "has_warnings": bool,                  # 是否有容差内 WARN
    "affected_row_count": int,             # 受容差影响的行数
    "total_rows": int,                     # 对齐行数
    "exact_match_rows": int,               # 精确匹配行数
    "tolerance_match_rows": int,           # 容差匹配行数
    "decision_reason": str,                # CRE 判定理由
    "warnings": list,                      # 容差 WARN 详情
    "mismatched_buckets": int,             # 不一致桶数
    "error_message": str,                  # 错误/警告信息
}
```

### 7.4 Shadow 异常安全

- 非检查异常全部捕获 → `diagnostic_available=False`, `human_review_recommended=True`
- exception 原因保留在 `error_message` 中
- **不改变** legacy status、修复路由、REVIEW_READY 门禁

### 7.5 EnvironmentManifest 与时区传递（v2.2 硬化）

`EnvironmentManifest` 已从最小模型（仅 3 个 SpecialFloatStrategy）扩展为完整严格 Pydantic 模型：

```python
class EnvironmentManifest(StrictModel):
    """环境清单——Pipeline 必须显式构造并传入，禁止 CRE 内部默认猜测。"""
    duckdb_version: str = ""                         # DuckDB 版本（审计追溯）
    spark_version: str = ""                          # PySpark 版本（审计追溯）
    timezone: str = ""                               # 时区标识（如 "Asia/Shanghai"）
    ansi_sql: bool = True                            # ANSI SQL 模式
    case_sensitive_compare: bool = False             # 大小写敏感比较
    decimal_strategy: DecimalStrategy = EXACT        # Decimal 策略（EXACT/QUANTIZE）
    null_strategy: NullStrategy = EQUAL              # NULL 策略（EQUAL/MISMATCH）
    nan_handling: SpecialFloatStrategy = HUMAN_REVIEW
    pos_inf_handling: SpecialFloatStrategy = HUMAN_REVIEW
    neg_inf_handling: SpecialFloatStrategy = HUMAN_REVIEW
```

- Pipeline 在 `_do_spark_physical_verify()` 中显式构造 `EnvironmentManifest` 并传入 `verify()`
- `_shadow_cre_diagnose` 接收 `EnvironmentManifest` 对象（非 dict），禁止内部默认构造
- 默认值均为保守安全策略（HUMAN_REVIEW for NaN/Inf、EXACT for Decimal、EQUAL for NULL）

### 7.6 CreShadowReport 严格模型（v2.2 新增）

Shadow 报告从 `dict[str, Any]` 升级为严格 Pydantic `CreShadowReport`（`extra="forbid"`）：

```python
class CreShadowReport(StrictModel):
    diagnostic_available: bool = False
    contract_hash: str = ""
    snapshot_id: str = ""
    cre_status: str = ""                 # CRE 原始状态
    mapped_status: str = ""              # 映射后 legacy 状态
    legacy_status: str = ""              # 生产状态
    status_consistent: bool = False
    human_review_recommended: bool = False
    has_warnings: bool = False
    warnings: list[CreShadowWarning] = []
    total_rows: int = 0
    exact_match_rows: int = 0
    tolerance_match_rows: int = 0
    affected_row_count: int = 0
    mismatched_bucket_count: int = 0
    decision_reason: str = ""
    error_message: str = ""
```

### 7.7 Singleton 对齐（v2.2 新增）

无 primary_keys 时的对齐策略：

- **双侧各 1 行** → singleton 对齐允许（跳过 KeyBasedRowAligner 的 PK gate，直接逐行比较）
- **任一侧多行** → NOT_EXECUTED + HUMAN_REVIEW 建议，禁止按行号或全部列猜键
- Singleton 模式下用占位主键 `__singleton__` 绕过 DecisionEngine PK gate

### 7.8 Audit Artifact（v2.2 新增）

每次 `verify()` 调用将 CRE shadow 结果写入 `cre_shadow_<report_id>.json`（位于 snapshot_dir）：

- 汇总指标：执行数、可用率、状态一致率、WARN 率、NOT_EXECUTED 率
- 分歧样本：status_consistent=False 时记录 CRE 与 legacy 状态差异
- 失败不阻断主流程——audit 写入失败仅记录日志

## 9. 残余风险

| 编号 | 风险说明 | 等级 | 说明 |
|------|---------|:----:|------|
| R-CRE-1 | **已解决**：真实 Dual-Engine 聚合执行已验证（AVG double 尾差、SUM decimal 精确匹配、GROUP BY 多行） | 低 | test_cre_dual_engine.py 覆盖 16 个真实执行用例（含时间戳、微秒、时区） |
| R-CRE-2 | **二级排序稳定化未实现**：非唯一主键的组内全行排序当前在 `encode_row` 层尚未显式实现 | 低 | 现有数据都保证主键唯一性，如出现非唯一主键需追加排序 |
| R-CRE-3 | **已解决**：timestamp 微秒级 + 非 UTC 时区（Asia/Tokyo/Europe/London）已在双引擎 Harness 中验证 | 低 | 3 个真实时区用例全部通过 |
| R-CRE-4 | **Bucket 性能未测量**：>50K 行大数据集的分桶均匀性和性能未测量 | 低 | 原型阶段不涉及性能优化 |
| R-CRE-5 | **VARCHAR 无 Unicode 归一化**：相同语义的不同 Unicode 编码（如 NFC/NFD）会被判定为不同 | 低 | 双引擎 VARCHAR 值通常来自同一上游，同字符串的 Unicode 编码一致 |

## 12. 接入前提与准入标准

### 12.1 准入标准（v2.3——零容忍）

正式接入生产 `verify()`（Phase 3——CRE 接管生产结论）的准入门槛：

1. **可执行样本状态一致率 100%**：所有 `diagnostic_available=True` 的样本，`status_consistent=True`
2. **零容忍错误检出率 100%**：CRE 正确检出所有已知差异场景（无假阴性）
3. **CRE→CONSISTENT 但 legacy→MISMATCH 样本数 = 0**：CRE 判断一致但生产判断不一致——零容忍
4. **NOT_EXECUTED 单独计算覆盖率**：因缺 PK/缺 output_columns 跳过的样本不计入一致率统计
5. **WARN 率仅诊断**：`CONSISTENT_WITH_WARN` 比例不作为准入门槛——仅用于诊断双引擎浮点尾差严重度
