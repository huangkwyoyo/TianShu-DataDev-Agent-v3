## 1. 背景与动机

### 1.1 当前问题

`PhysicalVerifier.verify()` 的全量比较方式存在以下问题：

1. **数据传输开销大**：DuckDB 和 Spark 全量 `output_rows` 通过网络传输到 Python 进程
2. **比较粒度粗**：只知道"不一致"，不知道"哪些行、哪些列"
3. **无可容忍差异机制**：浮点尾差（1e-15 级别）直接判为 MISMATCH
4. **架构耦合**：比较逻辑（~120 行 `_compute_diffs`）与验证流程耦合在同一方法中
5. **CRE 过渡期脚手架臃肿**：~2400 行过渡代码，包括自定义二进制编码协议、shadow 双轨制、7 层决策矩阵等

### 1.2 根因统计

从修复记录和诊断结果：

| 差异类型 | 占比 | 根因 |
|---------|:----:|------|
| float/double 末位差异（~1e-15） | **~92%** | 双引擎 IEEE 754 运算顺序差异 |
| Decimal 尾随零（`1.70` vs `1.7`） | ~5% | 字符串表示差异 |
| NULL 键缺失（Spark toJSON 省略 null） | ~3% | JSON 序列化差异 |

### 1.3 设计目标

**"引擎内部计算摘要，外部仅比较摘要"**——每个引擎只返回 `row_count + full_digest + samples`，比较层仅做 count/digest 布尔判断。

### 1.4 不可变边界与本轮范围

**以下边界不可改变**（后续任何实施必须遵守）：

1. **`row_count` 和 `full_digest` 始终基于全量结果**——不得因结果集大小切换为部分行 digest
2. **`samples` 不参与一致性判定**——仅用于人工确认数据形态，不能冒充差异行或替代全量 digest
3. **异常状态不得降级为 WARN**——超出资源限制、CDP 不支持、引擎失败、spec/snapshot 不一致等场景必须返回 `NOT_EXECUTED` / `UNSUPPORTED_SEMANTICS` / `HUMAN_REVIEW`
4. **CDP 版本化**——两侧引擎必须使用同一 CDP 版本和同一 `digest_spec_hash`，否则不可比
5. **`CONSISTENT_SAMPLE` 不宣称业务正确**——只证明"同一不可变快照上，受支持规范下的输出结果一致"

**本轮范围**：
- 冻结 CRE v3 设计文档（本文件）
- 明确 CDP v1 规范、过渡设计识别、准入标准
- **本轮不实施任何代码**——Phase 3a（CDP 编译器 + 黄金向量）为后续工作

## 2. 目标终态

### 2.1 核心流程

```
DuckDB 引擎内部                      Spark 引擎内部
  ├─ row_count: int        ←──→       ├─ row_count: int
  ├─ full_digest: SHA-256  ←──→       ├─ full_digest: SHA-256
  └─ pk_samples: list[dict] ←──→      └─ pk_samples: list[dict]

                     ↓ 只返回这三个值
         count1 == count2?  →  No  →  DIFFERENT（行数不同）
                     ↓ Yes
         digest1 == digest2? →  No  →  DIFFERENT + WARN
                               ↓        samples 仅供人工确认数据形态
                     ↓ Yes
              CONSISTENT_SAMPLE
```

### 2.2 核心原则

1. **引擎内部独立计算**：count / digest / samples 在各引擎内完成，不依赖 Python 中间层拉取全量数据
2. **比较层只做三件事**：比 count、比 digest、不同时用 samples 提供诊断线索
3. **没有 Python 逐行容差比较**：浮点/Decimal/Timestamp/NULL 规范由 CDP 定义并下推到引擎
4. **不一致 → WARN + 人工 review**：任何不一致都 WARN 预警，不自动判定"可接受"
5. **samples 不参与判定**：samples 只提供确定性诊断线索，不能冒充"差异行"

### 2.3 两引擎必须使用同一 CDP

不能直接比较 DuckDB/Spark 各自原生 `CAST + CONCAT_WS + SHA256` 的结果。两个引擎必须执行同一份版本化 **Canonical Digest Protocol（CDP）**，否则 NULL、Decimal、Timestamp、NaN、负零、分隔符转义和无序聚合都可能产生假差异。

## 3. 场景覆盖分析

### 3.1 完全覆盖（无需额外处理）

| 场景 | 覆盖机制 |
|------|---------|
| 行数一致 + 数据完全一致 | count 相同 + digest 相同 → CONSISTENT_SAMPLE |
| 行数不一致 | count 不同 → DIFFERENT |
| 确定性内容差异（非浮点列） | digest 不同 → DIFFERENT + samples 定位 |
| 列缺失（一方缺列） | schema/spec 校验阶段即进入 `HUMAN_REVIEW`——不应等到行 digest 后才暴露 |
| 列顺序不同 | Contract `output_columns` 定义 CDP 列序，双方遵循同一顺序 |
| 空结果集（双方 0 行） | count=0，`full_digest` 按 §6.2.4 完整计算（含 256 个空桶 digest）——**不使用空字符串或简化公式** |
| 单行结果 | 同多行，无特殊处理 |

### 3.2 SQL 层配合后覆盖（CDP 定义归一化规则）

| 场景 | 归一化策略 |
|------|-----------|
| 浮点尾差（AVG 差异） | CDP 明确 float_scale 时才允许 `ROUND(x, N)`，不能硬编码精度 |
| Decimal 表示差异 | CDP 固定 scale + 舍入模式，保留精度语义 |
| Timestamp 时区差异 | CDP 统一到声明时区和固定 ISO-8601 精度 |
| NULL 表示差异 | 使用独立 NULL 标签，禁止编码为空字符串 |
| 列名大小写差异 | Contract `output_columns` 定义权威列序 |

**关键洞察**：float/Decimal/timestamp/NULL 的归一化**不是比较层的事**——它是 CDP 编译器的责任。编译器生成引擎表达式后，比较层永远只看到一致的数据。

### 3.3 需降级处理

| 场景 | 发生概率 | 策略 |
|------|:---:|------|
| 有主键 + 结果集可控 | 最常见 | 全量：ORDER BY pk（用于 samples）→ COUNT + 多重集 full_digest（按 row_hash 聚合，不依赖 PK） |
| 有主键 + 结果集很大 | 中 | full_digest 仍覆盖全量——通过分桶 + 流式聚合控制峰值内存。samples 限制为 5~100 行。超资源上限 → NOT_EXECUTED |
| 无主键 | 低 | 多重集 digest 不依赖 PK。samples 使用 row_hash 排序取前 N 行。诊断能力降低，产生 WARN |
| 嵌套类型（STRUCT/LIST/MAP） | 极低 | `UNSUPPORTED_SEMANTICS`——不得回退引擎原生字符串后自动判一致 |
| BINARY/BLOB 列 | 极低 | v1 默认 UNSUPPORTED_SEMANTICS |

> **关键约束**：`row_count` 和 `full_digest` 始终覆盖全量结果。只有 `samples` 限制行数（5~100 行）。大结果集通过分桶、流式聚合和超时控制资源，**不改成部分行 digest**。超出资源限制时返回 `NOT_EXECUTED` 或 `HUMAN_REVIEW`，不能用采样一致替代全量摘要。

### 3.4 明确不覆盖——进入 NOT_EXECUTED / UNSUPPORTED_SEMANTICS / HUMAN_REVIEW

以下场景不能仅靠三个值自动判定，不得降成 WARN 后继续判定一致：

| 场景 | 判定状态 | 原因 |
|------|---------|------|
| CDP 不支持的类型（STRUCT/LIST/MAP/BINARY） | `UNSUPPORTED_SEMANTICS` | CDP v1 不覆盖嵌套/二进制类型 |
| subquery step 类型 | `UNSUPPORTED_SEMANTICS` | 尚无等价对比规则 |
| 缺失列类型、时区或浮点规范 | `HUMAN_REVIEW` | 无法生成相同 canonical bytes |
| 任一引擎未执行、执行失败、超时 | `NOT_EXECUTED` | 无有效摘要可供比较 |
| 摘要协议版本或 schema digest 不一致 | `HUMAN_REVIEW` | 两侧使用了不同 CDP 版本 |
| 需要定位具体差异单元格 | 启动 bucket drill-down | 这是诊断请求，不属于自动判定范畴 |

### 3.5 覆盖度总结（设计目标——未经验证）

> **以下为设计覆盖目标，并非实测数据。CDP v1 尚未实现，无跨引擎黄金向量，无性能基准。**

| 维度 | 设计覆盖 | 验证状态与说明 |
|------|:------:|------|
| 当前模板类型 | **目标 100%** | 全部是标量类型。待 CDP 编译器实现后用真实数据验证 |
| 当前差异根因 | **目标 100%** | float/Decimal/NULL——待 CDP 引擎表达式黄金向量确认 |
| 非标量类型 | **0%** | CDP v1 不覆盖，返回 UNSUPPORTED_SEMANTICS |
| 有主键 + 小结果集 | **目标 100%** | 全量 digest——待 CDP 编译器 + 双引擎黄金向量 |
| 有主键 + 大结果集 | **目标覆盖** | 分桶全量 digest——待性能基准（内存、耗时上限） |
| 无主键 | **降级覆盖** | 多重集 digest 不依赖 PK；samples 为 row_hash 排序行。诊断能力降低 |
| 资源安全 | **设计约束** | 超资源上限 → NOT_EXECUTED 或 HUMAN_REVIEW，不降级为采样 digest |

## 4. CRE v2 现状架构（过渡期）

### 4.1 当前实际数据流

```
verify()
  │
  ├─ Step 2: DuckDB 执行 SQL → SparkExecutionResult
  │    └─ output_rows: list[dict]  ← 全量行数据拉取到 Python
  │
  ├─ Step 3: Spark 执行 PySpark DSL → SparkExecutionResult
  │    └─ output_rows: list[dict]  ← 全量行数据拉取到 Python
  │
  ├─ Step 4: ResultCanonicalizer.canonicalize()
  │    └─ Python 侧全量行排序 + 归一化
  │
  ├─ Step 7: _compute_diffs()
  │    └─ Python 逐行逐列 for 循环比较
  │
  ├─ Step 8: Legacy 判定（diff_count==0 → RESULT_CONSISTENT / MISMATCH）
  │
  └─ _shadow_cre_diagnose()（shadow 模式）
       ├─ CREEncoder 逐行编码（自定义 12 种类型标签二进制协议）
       ├─ KeyBasedRowAligner PK 对齐
       ├─ ToleranceComparator 逐列容差比较
       ├─ DecisionEngine 7 Gate 判定
       └─ 映射 CRE 状态 → legacy 状态 → 比较 status_consistent
```

### 4.2 核心差距

| # | 目标 | 当前实现 | 严重度 |
|---|------|---------|:---:|
| 1 | 引擎只返回 count+digest+samples | 两个引擎都返回**全量行数据**（`output_rows`） | **致命** |
| 2 | digest 在引擎内部计算 | digest 在 Python 侧由 CREEncoder 计算（自定义二进制协议，~400 行） | **致命** |
| 3 | count+digest 一致即判定 | 两套比较系统并行：legacy `_compute_diffs()` + CRE shadow | **冗余** |
| 4 | 不一致时只 WARN+人工 review | DecisionEngine 7 层 Gate 尝试自动区分 MISMATCH/HUMAN_REVIEW | **过度** |
| 5 | PK 样本用于诊断定位 | `sample_rows = duckdb_rows[:5]`——取排序后前 5 行，不是 PK 样本 | **错误** |

## 5. 过渡设计识别

### 5.1 逐项识别

以下按"距离终态的差距"从大到小排列，共 10 项，总计约 **2200~2400 行**可简化/删除/移动。

---

#### TD-0A：`output_rows` 全量回传（最高优先级）

**当前设计**：DuckDB 和 Spark 都把完整 `output_rows: list[dict]` 拉回 Python。

**为什么是过渡设计**：直接违反"引擎内部计算、外部只返回摘要"的目标，使 Python 内存、序列化成本和结果规模线性增长。

**终态动作**：
1. 执行器新增摘要执行入口，直接返回 `EngineDigestSummary`
2. PhysicalVerifier 主路径不再接收完整 `output_rows`
3. `output_rows` 仅允许存在于有明确上限的调试/测试 API
4. 在双引擎 CDP 黄金测试通过前保留旧路径 shadow，不得直接删除后切换结论

---

#### TD-0B：`ResultCanonicalizer` Python 侧排序和归一化（~200 行）

**当前设计**：Python 拉取全量行后排序并逐值规范化。

**为什么是过渡设计**：规范计算应该由 CDP 编译器生成引擎表达式，在引擎内执行。

**终态动作**：删除 Python 全量 canonicalizer；保留小型 `CreDigestSpecValidator`，只负责检查 CDP 配置完整性和支持类型。

---

#### TD-0C：`_compute_diffs()` Python 逐行逐列比较（~120 行）

**当前设计**：Python 对两侧 canonical rows 执行逐行逐列循环并产生完整差异列表。

**为什么是过渡设计**：主结论只需要 count/digest。完整 cell diff 既要求回传全量结果，也不是自动门禁所必需。

**终态动作**：主路径删除。digest 不同直接产生 `DIFFERENT` + WARN。若用户主动请求诊断，执行独立 bucket drill-down。

---

#### TD-1：CRE2 自定义二进制编码协议（~400 行）

**当前设计**：自定义 12 种类型标签的二进制编码格式（CRE2 magic、big-endian 补码、IEEE 754 raw bytes、Decimal length-prefix 等），每行编码后计算 SHA-256。

**为什么是过渡设计**：
- 终态中 digest 在引擎内部计算——DuckDB `sha256()`、Spark `sha2()` 比 Python 侧自定义编码快几个数量级
- 自定义编码的核心诉求"确定性"终态由 CDP 引擎表达式实现
- 二进制编码的复杂性与目标场景不匹配——我们只需要比较结果是否一致，不需要可逆的跨引擎序列化

**终态动作**：
1. 实现 CDP 编译器，分别生成 DuckDB 与 Spark canonical 表达式
2. 现有编码器仅作为测试 oracle——验证引擎表达式，不作为生产 fallback
3. 引擎无法表达的类型返回 `UNSUPPORTED_SEMANTICS`

---

#### TD-2：Shadow 双轨制（~200 行模型定义）

**当前设计**：CRE 以 shadow 模式运行——同时计算 CRE 和 legacy 结论，比较一致性（`status_consistent`）。

**为什么是过渡设计**：
- Shadow 的目的是验证 CRE 正确性——一旦零容忍准入通过，使命完成
- `mapped_status`、`legacy_status`、`status_consistent`、`human_review_recommended` 全部是"与旧系统对比"的元数据——旧系统下线后无意义
- 终态状态：`CONSISTENT_SAMPLE` / `DIFFERENT` / `NOT_EXECUTED` / `UNSUPPORTED_SEMANTICS` / `HUMAN_REVIEW`

**终态动作**：
1. CRE 接管生产结论后，删除 shadow 双轨
2. 删除 `CreShadowReport` 中的 `mapped_status`/`legacy_status`/`status_consistent`/`human_review_recommended`
3. 重命名 `CreShadowReport` → `CreComparisonResult`
4. 删除 `_CRE_TO_LEGACY_MAP` 映射表

---

#### TD-3：EnvironmentManifest 运行时策略分支（~150 行）

**当前设计**：11 字段 + 4 枚举（DecimalStrategy、NullStrategy、SpecialFloatStrategy×3），比较层 7 Gate 分支读取这些策略。

**为什么需要缩减而不是删除**：
- 策略应收口到版本化 CDP，不应散落在运行时判定分支
- 但引擎版本、时区、ANSI、大小写等**审计信息**不能删除——它们记录执行环境，属于 Execution Boundary 要求
- 比较层不应再读取这些字段做策略分支，只验证 manifest 与 CDP 是否兼容

**终态动作**：
1. canonical 行为迁移到 `CreDigestSpec.protocol_version` 对应的 CDP 定义
2. EnvironmentManifest 保留引擎版本、时区、ANSI、大小写等审计字段
3. 比较层不再读取这些字段做分支

---

#### TD-4：ToleranceComparator 逐列容差比较（~350 行）

**当前设计**：先尝试 exact digest 匹配，不匹配时逐列执行 `math.isclose`，区分 `WITHIN_TOLERANCE` vs `OUT_OF_TOLERANCE`。

**为什么是过渡设计**：
- 终态中 digest 相同 → 一致；digest 不同 → 不一致。不存在"digest 不同但容差内可接受"的中间状态
- 浮点容差应由 CDP 在引擎内执行一致的 ROUND——只有 Contract 明确允许 scale-based 比较时才使用

**终态动作**：
1. 浮点规范写入 CDP 并下推引擎
2. 删除 `ToleranceComparator.compare_row()` 的逐列容差比较
3. 删除 `ToleratedDifferenceWarning`、`ToleratedFieldDetail`、`WITHIN_TOLERANCE`/`OUT_OF_TOLERANCE` 分类

---

#### TD-5：DecisionEngine 7 层 Gate 矩阵（~250 行）

**当前设计**：7 层 Gate（缺配置→HUMAN_REVIEW、对齐失败→HUMAN_REVIEW、重复键→HUMAN_REVIEW、行数不匹配→MISMATCH、不可容忍类型→MISMATCH、超容差→MISMATCH、未知原因→MISMATCH/HUMAN_REVIEW）。

**为什么是过渡设计**：
- 成功摘要的判定逻辑极简：count 不同 → `DIFFERENT`；digest 不同 → `DIFFERENT`；都相同 → `CONSISTENT_SAMPLE`
- Gate 1-3（配置/对齐/重复键检查）是**前置条件**，应在引擎执行前完成
- Gate 5-7 在 digest 模式下全部被"digest 不同"一个条件覆盖

**终态动作**：
1. 成功摘要简化为 count/digest 判定
2. 前置条件移到 `verify()` 入口——在执行引擎前检查
3. `decision_reason` 只保留结构化 reason code：`COUNT_DIFFERENT`、`DIGEST_DIFFERENT`、`SUMMARY_CONSISTENT`、`UNSUPPORTED_CDP`、`ENGINE_NOT_EXECUTED`

---

#### TD-6：CreHarnessAggregation 跨请求聚合（~150 行）

**当前设计**：聚合多个样本，计算一致率、假阴性率，通过 `passes_admission` 判定准入标准。

**为什么是过渡设计**：
- 这是 Harness（测试框架）的功能，不是 CRE 比较机制本身
- `golden_label`、`false_negative_rate` 是测试场景概念——回答"CRE 准不准"，不是"数据一不一致"
- 准入标准应由 Harness 测试套件定义，不应硬编码在生产代码中

**终态动作**：移至 `tests/harness/`，删除 `passes_admission` 属性，保留统计计算逻辑。

---

#### TD-7：ReviewPackageFinalizer 原子追加（~380 行）

**当前设计**：验证已有 Package → 原子写入 `cre_shadow_report.json` → 更新 provenance。

**为什么是过渡设计**：
- 终态中 CRE 结论**本身就是**验证结果的核心——不需要"追加"到已有的 SQL Review Package
- 380 行的哈希验证 + 原子写入逻辑，本质是"安全放文件"——应该直接由 packager 写入

**终态动作**：
1. `ReviewPackageBuilder` 直接包含 CRE 比较结果
2. 删除 `ReviewPackageFinalizer` 整个类
3. `FinalizerResult` → 简单的 `(success: bool, path: str, hash: str)` tuple

### 5.2 过渡设计对照表

| 编号 | 过渡组件 | 行数（估） | 过渡原因 | 终态去向 | Phase 3 动作 |
|------|---------|:---:|---------|---------|-------------|
| TD-0A | `output_rows` 全量回传 | 数据通路 | 摘要应在引擎内完成 | `EngineDigestSummary` | 新增摘要执行入口 |
| TD-0B | ResultCanonicalizer | ~200 | Python 全量排序/归一化 | CDP 引擎表达式 | 删除行处理，保留 spec validator |
| TD-0C | `_compute_diffs()` | ~120 | Python 逐行逐列比较 | digest + 可选 bucket drill-down | 从生产主路径删除 |
| TD-1 | CRE2 二进制编码 | ~400 | digest 应在引擎侧计算 | CDP DuckDB/Spark 编译器 | Python 编码器仅作测试 oracle |
| TD-2 | Shadow 双轨制 | ~200 | 验证 CRE 正确性的临时机制 | 无（删除） | 删除 mapped/legacy/consistent 字段 |
| TD-3 | EnvManifest 运行时策略分支 | ~150 | canonical 行为应版本化 | 保留完整审计；行为收口 CDP | 不直接删除宪法要求字段 |
| TD-4 | ToleranceComparator | ~350 | digest 比较替代逐列容差 | CDP 浮点规范 + 可选诊断 | 删除生产逐列比较 |
| TD-5 | DecisionEngine 7 Gate | ~250 | 成功摘要只需 count/digest | 结构化状态薄层 | 保留未执行/不支持边界 |
| TD-6 | HarnessAggregation | ~150 | 测试工具混入生产代码 | 移到 tests/harness/ | 移动文件，不删除 |
| TD-7 | ReviewPackageFinalizer | ~380 | Shadow 事后追加 | ReviewPackageBuilder 直接包含 | 合并入 packager |

## 6. Canonical Digest Protocol（CDP v1）

CDP 是一个小型、版本化的摘要协议。它不是 Python 编码主路径，而是 DuckDB 和 Spark 摘要表达式的**共同规格**。两侧引擎必须产生完全相同的 SHA-256 输入字节序列。

### 6.1 Spec 定义

```python
class CreDigestSpec(StrictModel):
    """CDP v1 摘要规格——定义 full_digest 计算所需的所有列级配置。冻结后不可变。"""
    protocol_version: Literal["cdp-v1"] = "cdp-v1"
    output_columns: list[str]             # 严格有序
    type_families: list[TypeFamily]       # 封闭枚举，不含 NULL——NULL 是值状态，不是列类型
    timezone: str                         # 如 "Asia/Shanghai"
    decimal_precision: list[int | None]   # DECIMAL 列的 precision；非 DECIMAL 列为 None
    decimal_scale: list[int | None]       # DECIMAL 列的 scale；非 DECIMAL 列为 None
    float_precision: list[int | None]     # FLOAT 列的 ROUND 精度；None=该列不归一化
    # CDP v1 固定 256 桶——bucket_id 为 row_hash 首字节（0x00-0xFF）
    _BUCKET_COUNT: ClassVar[int] = 256

class CreSampleSpec(StrictModel):
    """CDP v1 采样规格——仅影响 samples 输出，不影响 full_digest。"""
    primary_keys: list[str]               # 可空；用于 samples 分层和诊断定位
    sample_size: int = 5                  # samples 行数上限

class TypeFamily(Enum):
    """列声明类型——封闭集合。NULL 不是 TypeFamily，是值级编码。"""
    BOOLEAN = 0x01
    INT8 = 0x02
    INT16 = 0x03
    INT32 = 0x04
    INT64 = 0x05
    FLOAT32 = 0x06
    FLOAT64 = 0x07
    DECIMAL = 0x08
    VARCHAR = 0x09
    DATE = 0x0A
    TIMESTAMP = 0x0B
```

**`digest_spec_hash` 计算**——仅包含影响 full_digest 的字段（列名、类型、精度、时区、桶数）。`primary_keys` 和 `sample_size` 属于 `CreSampleSpec`，不参与 digest 计算。

使用确定性 canonical JSON（按 key 字典序、无空白、UTF-8）：

```
canonical_json =
    '{"bucket_count":256,'
    + '"columns":['
    + '{"float_precision":' + fp_0 + ',"decimal_precision":' + dp_0
    + ',"decimal_scale":' + ds_0 + ',"name":"' + col_0 + '","type":"' + type_0 + '"},'
    + ...
    + '],'
    + '"protocol_version":"cdp-v1",'
    + '"timezone":"Asia/Shanghai"}'
digest_spec_hash = SHA256(canonical_json)  # 32 字节原始 SHA-256 输出
```

**关键约束**：
- `canonical_json` 中 `float_precision`/`decimal_precision`/`decimal_scale` 为 `null`（JSON null）或整数
- `primary_keys`、`sample_size` **不出现在** canonical JSON 中——它们属于 `CreSampleSpec`，改变采样策略不应改变 full_digest
- `digest_spec_hash` 是 32 字节原始 SHA-256 输出，**直接写入** `full_digest_input`，不再经过二次哈希

### 6.2 CDP v1 编码规则（冻结）

> **以下规则是 CDP v1 规范本身。黄金向量用于验证实现是否正确，不用于决定规范语义。**

#### 6.2.1 字段级规范化

| 类型 | 编码规则（CDP v1 冻结定义） | 黄金向量验证项 |
|------|---------|------|
| **NULL 值** | 任何声明类型的字段值为 SQL NULL 时：`tag`=声明类型，`length_prefix`=`0xFFFFFFFF`（4 字节全 1），无 `value_bytes`。与空字符串、零值严格区分 | DuckDB/Spark 对 NULL 的检测行为 |
| **BOOLEAN** | `"true"` / `"false"`（小写 UTF-8） | — |
| **INT8/16/32/64** | 十进制文本，无千位分隔符，无前导零（`0` 本身除外），负号 `-` 前缀 | 负零（应不存在） |
| **FLOAT32/64** | `float_precision` 非 None → `ROUND(x, N)` 后编码，**舍入模式固定为 ROUND_HALF_UP（远离零方向）**：正数 .5 向上（2.5→3），负数 .5 远离零（-2.5→-3）。NaN→`"nan"`、+Inf→`"inf"`、-Inf→`"-inf"`、负零→`"-0.0"`、正零→`"0.0"`。所有 token 为固定小写 ASCII | 各引擎 ROUND_HALF_UP 实现一致性；NaN/Inf 检测 |
| **DECIMAL** | `ROUND_HALF_UP`（远离零）舍入到 `decimal_scale` 位：正数 .5 向上（1.5→2），负数 .5 远离零（-1.5→-2）。编码为紧邻十进制文本：`[-]unscaled_int`（无小数点、无前导零） | 引擎 `ROUND_HALF_UP` 支持；precision>38 |
| **VARCHAR** | UTF-8 原始字节（不附加 NFC/NFD 归一化）。长度由字段级 `length_prefix` 统一管理，不在类型编码内重复添加长度前缀 | Unicode 归一化需求 |
| **DATE** | `"YYYY-MM-DD"`（ISO-8601，固定 10 字节 UTF-8） | — |
| **TIMESTAMP** | 转换为声明时区后编码：`"YYYY-MM-DDTHH:MM:SS.ffffff+HH:MM"`（ISO-8601 扩展格式，微秒精度，固定 32 字节 UTF-8）。naive datetime（无时区）→ 拒绝执行（`HUMAN_REVIEW`），禁止猜测时区。**DST 重叠时间**（fall-back，同一本地时间出现两次）和**不存在时间**（spring-forward，本地时间被跳过）→ `HUMAN_REVIEW`，不进行时区转换 | 各引擎时区转换行为；DST 边界 |
| **嵌套/二进制** | v1 硬拒绝 `UNSUPPORTED_SEMANTICS` | — |

#### 6.2.2 行级编码

```
row_canonical_bytes =
    field_1_tag  || field_1_length_prefix  || field_1_value_bytes ||
    field_2_tag  || field_2_length_prefix  || field_2_value_bytes ||
    ...
    field_n_tag  || field_n_length_prefix  || field_n_value_bytes

row_hash = SHA256(row_canonical_bytes)
```

| 字节段 | 长度 | 编码 |
|--------|------|------|
| `tag` | 1B | TypeFamily 枚举值（0x01-0x0B）。值 NULL 时仍使用列的声明类型，通过 length_prefix 区分 |
| `length_prefix` | 4B BE | `0xFFFFFFFF` → 值为 NULL（无 value_bytes 跟随）。`0x00000000`-`0xFFFFFFFE` → value_bytes 字节数 |
| `value_bytes` | 变长 | 按 §6.2.1 规范编码的 UTF-8 字节。仅 length_prefix ≠ `0xFFFFFFFF` 时存在 |

#### 6.2.3 桶级聚合

```
bucket_id = row_hash[0]  (1 byte, 0..255)
桶内聚合（先 GROUP BY row_hash 去重计数，再按 row_hash 排序）：
    bucket_payload = bucket_id (1B)
                  || unique_hash_count (8B BE)
                  || (row_hash_0 || occurrence_count_0 (8B BE))
                  || (row_hash_1 || occurrence_count_1 (8B BE))
                  || ...
bucket_digest = SHA256(bucket_payload)
```

- `row_hash` 按字典序排序（unsigned big-endian 32-byte 比较）
- `occurrence_count`（8B BE）保留重复行的次数——确保多重集语义。两行相同值但不同行号 → 同一个 row_hash + count=2。8 字节支持 up to 2^64-1 次重复——`full_comparison` 模式下不会成为瓶颈
- 即使所有行落到同一个桶，每个 distinct row_hash 只存一次 32B hash + 8B count

#### 6.2.4 最终 full_digest

```
full_digest_input =
    protocol_version (UTF-8 "cdp-v1") ||   # 6 bytes
    digest_spec_hash (32B) ||               # 直接使用 §6.1 的 32 字节 SHA-256 原始输出——不再二次哈希
    total_row_count (8B BE) ||              # COUNT(*) 结果
    bucket_count (2B BE, fixed 0x0100) ||   # 256
    bucket_0_id (1B) || bucket_0_digest (32B) ||
    bucket_1_id (1B) || bucket_1_digest (32B) ||
    ...                                     # 256 个桶全部出现——空桶 digest=SHA256(empty_payload)
    bucket_255_id (1B) || bucket_255_digest (32B)

full_digest = SHA256(full_digest_input)
```

- `digest_spec_hash` 直接作为 32 字节原始值写入，不含 `primary_keys` 和 `sample_size`——这两个字段属于 `CreSampleSpec`，改变采样策略不影响 full_digest。**CDP 列级配置任何字段变更必然导致 full_digest 变化。**
- 总字节数 = 6 + 32 + 8 + 2 + 256 × (1 + 32) = **8496 bytes**（恒定，不受行数影响）
- 空桶 payload = `bucket_id || 0x0000000000000000`（unique_hash_count=0，8B BE），空桶 digest = SHA256(空桶 payload)

#### 6.2.5 黄金向量验证

**黄金向量验证规范，不定义规范。** CDP v1 规则如上述定义——黄金向量用于确认 DuckDB 和 Spark 的 CDP builder 实现正确：

- 空结果集（count=0）→ digest 必须 = SHA256(full_digest_input with total_row_count=0)
- 单行、重复行、各类型边界值、NULL 值行
- Float NaN/Inf/负零、Decimal 极大/极小值
- Timestamp 跨时区、DST 边界
- 至少一个真实模板完整结果集

**每个向量的 `full_digest` 在 DuckDB 和 Spark 上必须完全一致。不一致 → CDP builder 实现有 bug，不是规范有问题。**

### 6.3 返回载荷

```python
class EngineDigestSummary(StrictModel):
    """引擎摘要——仅 execution_status==SUCCESS 时填充"""
    row_count: int
    full_digest: str
    samples: list[dict]              # 确定性样本（不参与判定）

class DigestExecutionEnvelope(StrictModel):
    """执行外围——比较前必须验证。数据契约：status==SUCCESS → summary 非 None；status!=SUCCESS → summary=None"""
    execution_status: Literal["SUCCESS", "FAILED", "TIMEOUT"]
    snapshot_id: str                  # 输入数据快照标识
    digest_spec_hash: str             # SHA256(CreDigestSpec 的 canonical JSON)
    protocol_version: str             # "cdp-v1"
    engine_version: str               # DuckDB 版本 / PySpark 版本
    error: str | None                 # 失败原因（status!=SUCCESS 时填充）
    summary: EngineDigestSummary | None  # None when execution_status != SUCCESS
```

**比较前强制验证**：

```
spec_hash_d == spec_hash_s 且 snapshot_id_d == snapshot_id_s？
  → No：HUMAN_REVIEW（两侧使用了不同 CDP 或不同数据快照）
  → Yes：继续验证 execution_status 和 summary
execution_status 均为 SUCCESS 且 summary 均非 None？
  → No：NOT_EXECUTED（至少一侧未成功执行）
  → Yes：继续比较 row_count 和 full_digest
```

**两侧误用了不同 CDP/spec 时，比较层无法区分"数据不同"和"摘要协议不同"。必须通过 `digest_spec_hash` 和 `snapshot_id` 在校验阶段排除。**

### 6.4 样本策略

- 有主键：取 PK 最小的 N 行（按 CDP 编码后的 PK bytes 字典序排序，row_hash 为 tiebreaker）
- 无主键：取 row_hash 最小的 N 行
- N = `sample_size`（默认 5，上限 100）
- 样本用于人工确认数据形态——不能据此定位 digest 差异，也不能替代 bucket drill-down

## 7. 资源安全与大结果集

### 7.1 核心原则

**`row_count` 和 `full_digest` 始终覆盖全量结果。只有 `samples` 限制行数。**

```
row_count   → 始终 COUNT(*)——全量扫描，不可避免
full_digest → 始终覆盖全部行——通过分桶 + 流式聚合控制峰值内存
samples     → 限制为 5～100 行——仅用于人工确认数据形态，不参与判定
```

**大结果集不改成部分行 digest。** 超出资源限制时返回 `NOT_EXECUTED` 或 `HUMAN_REVIEW`，不能用采样一致替代全量摘要。

### 7.2 大结果集资源控制

digest 计算本身不需要 `ORDER BY` 全量数据——按 `row_hash` 前缀分桶即可：

1. 每行计算 `SHA256(CDP_canonical_bytes)` → `row_hash`（32 bytes）
2. 取 `row_hash[0]` → 256 个桶（CDP v1 固定，不可调）
3. 桶内 `GROUP BY row_hash` → `(row_hash, occurrence_count)` 对——重复行只存一份 hash
4. 桶内按 `row_hash` 排序 → `SHA256(bucket_id || unique_count || hash_0 || count_0 || ...)` → `bucket_digest`
5. 按 bucket_id 排序 → 拼接 → 最终 `full_digest`

**内存不是固定值**——实际取决于 distinct row_hash 的基数，而非行数：
- 100 万行全相同 → 1 个 distinct hash × (32+8)B ≈ 40B（极省）
- 100 万行全不同 → 100 万个 hash × (32+8)B ≈ 40MB（总 distinct hash 内存）
- 真正最坏：全部行 distinct 且落到同一桶 → 单个桶 100 万 × 40B ≈ **40MB**（远大于均匀分布的 ~156KB/桶）
- 重复行越多越省——`GROUP BY` 去重后只存 distinct hash
- `full_comparison` 模式下 `MAX_RESULT_ROWS` 放宽，distinct hash 可能远超 100 万——**实施前必须用真实数据做基准测试**，不在设计中承诺固定内存上限，基准测试结果作为 CDP builder 实现的性能门禁

### 7.3 资源上限约束

| 约束项 | 默认值 | 超限行为 |
|--------|--------|---------|
| `MAX_RESULT_ROWS` | **500 万** | 预估行数超过 → NOT_EXECUTED（不启动摘要计算） |
| `QUERY_TIMEOUT_SEC` | **30** | 单引擎查询超时 → NOT_EXECUTED |
| `MAX_SAMPLE_ROWS` | **100** | samples 最多取回的行数。不影响 digest 计算 |
| `full_comparison` | **False（默认）** | 显式指定后放宽 `MAX_RESULT_ROWS` 限制。用户自担风险 |

### 7.4 资源安全保证

| 场景 | 引擎内存上限 | Python 内存上限 | 预估耗时 |
|------|:-----------:|:--------------:|:-------:|
| 小结果集（≤10K 行）| ~5MB | ~2MB | ~2s |
| 中等结果集（10K~100K 行）| ~10MB | ~2MB | ~5s |
| 大结果集（100K~1M 行）| ~30MB（分桶） | ~2MB | ~15s |
| 超大结果集（>500 万行）| — | — | NOT_EXECUTED（预估拒绝） |
| 显式全量（>500 万行）| 取决于数据 | 取决于数据 | 用户自担风险 |

**辅助开发工具不会因为验证而把电脑搞崩。**

### 7.5 无主键方案

多重集 `full_digest` 不依赖主键——按 `row_hash`（CDP SHA-256）排序和分桶。主键仅用于 `samples` 提取和诊断定位。

无主键时：
- `full_digest` 按 row_hash 多重集聚合，完全确定、不依赖引擎返回顺序
- `samples` 取 row_hash 最小的 N 行（确定性、跨引擎一致——前提是 CDP 产生的 row_hash 相同）
- 诊断能力降低（样本行不能按业务主键追溯），产生 WARN 标记

### 7.6 有维度列时的分层 samples

当存在像 `pickup_date_key`、`borough` 这样的维度列时，samples 可按维度分层提取：

```sql
WITH sampled AS (
    SELECT *, ROW_NUMBER() OVER (
        PARTITION BY dimension_col
        ORDER BY CDP_pk_bytes, row_hash  -- PK 字节字典序 + row_hash tiebreaker，确保确定性
    ) AS rn
    FROM (original_query)
)
SELECT *
FROM sampled WHERE rn <= 5  -- 每层最多 5 行，用于人工确认
```

> **注意**：`ORDER BY dimension_col`（分区键本身）在分区内所有行的排序键完全相同，排序结果不确定。必须使用 CDP PK bytes（或 row_hash）作为排序键。

## 8. CRE v3 终态架构

### 8.0 自动判定前提条件链

**以下五个条件全部满足，方可自动判定 `CONSISTENT_SAMPLE` / `DIFFERENT`：**

```
条件 1: 同一 Snapshot（snapshot_id_d == snapshot_id_s）
        ├─ 强制位置：compare() 入口
        └─ 不满足 → HUMAN_REVIEW（不同数据快照，不可比）

条件 2: 同一 digest_spec_hash（CDP 版本 + Spec 完全一致）
        ├─ 强制位置：compare() 入口
        └─ 不满足 → HUMAN_REVIEW（不同摘要协议，不可比）

条件 3: CDP 支持全部输出类型（type_families ⊆ CDP v1 覆盖范围）
        ├─ 强制位置：CreDigestSpecValidator（引擎执行前）
        └─ 不满足 → UNSUPPORTED_SEMANTICS（拒绝执行，不进入 compare）

条件 4: 全量 row_count（COUNT(*) 覆盖全部结果行）
        ├─ 强制位置：引擎 CDP builder 实现 + 准入标准 Gate 6（大结果集全量覆盖测试）
        └─ 不满足 → NOT_EXECUTED（超资源上限拒绝）或 HUMAN_REVIEW

条件 5: 全量多重集 digest（SHA256 基于全部行 row_hash 分桶聚合）
        ├─ 强制位置：引擎 CDP builder 实现 + 准入标准 Gate 6
        └─ 不满足 → NOT_EXECUTED 或 HUMAN_REVIEW
```

**任一条件不满足 → 不得返回 `CONSISTENT_SAMPLE` 或 `DIFFERENT`。** 条件 1-2 在 `compare()` 中验证，条件 3 在 spec 校验阶段验证，条件 4-5 由引擎实现保证并通过准入测试验证。


### 8.2 比较层极简实现

```python
class CreComparisonResult:
    """CRE 比较结果——极简数据模型"""
    status: str                       # CONSISTENT_SAMPLE | DIFFERENT | NOT_EXECUTED | UNSUPPORTED_SEMANTICS | HUMAN_REVIEW
    duckdb_count: int
    spark_count: int
    count_match: bool
    digest_match: bool
    duckdb_digest: str
    spark_digest: str
    pk_samples_duckdb: list[dict]    # 用于诊断，不参与判定
    pk_samples_spark: list[dict]
    decision_reason: str             # COUNT_DIFFERENT | DIGEST_DIFFERENT | SUMMARY_CONSISTENT | SPEC_MISMATCH | ENGINE_NOT_EXECUTED

def compare(duckdb: DigestExecutionEnvelope, spark: DigestExecutionEnvelope) -> CreComparisonResult:
    """摘要比较——唯一判定逻辑。

    前置条件（不在本函数内验证，由上游保证）：
      - 条件 3（CDP 支持全部类型）：CreDigestSpecValidator 在引擎执行前已校验
      - 条件 4-5（全量 count + 全量 digest）：引擎 CDP builder 实现保证 + 准入 Gate 6 验证
    """

    # 条件 1-2：同一 Snapshot + 同一 CDP Spec
    if duckdb.digest_spec_hash != spark.digest_spec_hash:
        return CreComparisonResult(status="HUMAN_REVIEW", decision_reason="SPEC_MISMATCH", ...)
    if duckdb.snapshot_id != spark.snapshot_id:
        return CreComparisonResult(status="HUMAN_REVIEW", decision_reason="SNAPSHOT_MISMATCH", ...)

    # 条件 4-5（隐含）：引擎执行成功 + summary 非空 → row_count 和 full_digest 必定是全量的
    if duckdb.execution_status != "SUCCESS" or spark.execution_status != "SUCCESS":
        return CreComparisonResult(status="NOT_EXECUTED", decision_reason="ENGINE_NOT_EXECUTED", ...)
    if duckdb.summary is None or spark.summary is None:
        return CreComparisonResult(status="NOT_EXECUTED", decision_reason="ENGINE_NOT_EXECUTED", ...)

    # 条件 4：全量 COUNT 比较
    count_match = duckdb.summary.row_count == spark.summary.row_count
    if not count_match:
        return CreComparisonResult(
            status="DIFFERENT", count_match=False, digest_match=False,
            decision_reason="COUNT_DIFFERENT", ...
        )

    # 条件 5：全量 DIGEST 比较
    digest_match = duckdb.summary.full_digest == spark.summary.full_digest
    if not digest_match:
        return CreComparisonResult(
            status="DIFFERENT", count_match=True, digest_match=False,
            decision_reason="DIGEST_DIFFERENT", ...
        )

    # 五个条件全部满足
    return CreComparisonResult(
        status="CONSISTENT_SAMPLE", count_match=True, digest_match=True,
        decision_reason="SUMMARY_CONSISTENT", ...
    )
```

### 8.3 适用结论

`row_count + full_digest + samples` 足以覆盖大部分**关系型、标量列、同一不可变快照**的 SQL/Spark 结果一致性验证，前提是：

1. 两侧输出列集合、列顺序和类型族已经确定
2. 两侧使用同一 CDP 版本和同一 `CreDigestSpec`
3. 所有输出类型都在 CDP 支持范围内
4. 全量 digest 对结果的**多重集**进行承诺，重复行不能被忽略
5. 执行成功且摘要确实覆盖全量结果，而不是抽样结果

该方案证明的是"当前不可变快照上，受支持规范下的输出结果一致"，状态应映射为 `CONSISTENT_SAMPLE`，不能宣称业务正确、生产全量正确或上线批准。

## 9. 简化实施计划

> **writing-plans 必须严格按以下 Phase 顺序展开，每个 Phase 完成后验证其门禁条件，全部通过才能进入下一 Phase。禁止跨 Phase 并行或提前执行删除动作。**

### 9.1 实施阶段

#### Phase 3a —— CDP Oracle + 黄金向量（P0，不可并行）

**目标**：在 Python 侧实现一个独立于引擎的 CDP canonical serializer，作为测试 oracle，并建立跨引擎黄金向量集。

**交付物**：
- `CreDigestSpec` + `CreSampleSpec` 的 Pydantic 模型（已有设计）
- `CdpCanonicalSerializer`：纯 Python 实现 CDP v1 字段级编码 + 行级 row_hash + 桶级聚合 + full_digest。**不依赖任何 SQL 引擎**
- 黄金向量集：空结果、单行、重复行、各 TypeFamily 边界值、NULL、NaN/Inf/负零、Decimal 极值、Timestamp 跨时区/DST、列缺失/类型不匹配等故障场景
- 每个黄金向量的预期 `full_digest` 由 Python oracle 计算并作为参考值

**门禁**：Python oracle 对同一输入重复计算 10 次 → `full_digest` 100% 稳定；黄金向量集覆盖 §6.2.1 全部 TypeFamily + 全部故障场景。

| 阶段 | 动作 | 风险 | 优先级 |
|------|------|:---:|:---:|
| **Phase 3a** | 实现 Python CDP canonical serializer（oracle） + 建立黄金向量集 | 高——oracle 错误会导致后续全部误判 | **P0** |

#### Phase 3b —— 双引擎 CDP Builder + 属性测试（P0，依赖 3a）

**目标**：在 DuckDB 和 Spark 两侧分别实现 CDP digest builder（生成引擎原生表达式计算 row_hash → bucket → full_digest），以 shadow 模式返回 `DigestExecutionEnvelope`，不接管结论。

**交付物**：
- DuckDB CDP builder：生成 DuckDB SQL 表达式（`SHA256(...)` 嵌套调用）
- Spark CDP builder：生成 PySpark 表达式
- 双引擎属性测试：随机生成 `(rows, spec)` 对，验证 `engine.full_digest == python_oracle.full_digest`
- 性能基准测试：10K / 100K / 500K 行结果集的端到端耗时 + 峰值内存

**门禁**：属性测试 1000 组随机输入 → DuckDB ≡ Spark ≡ Python oracle（零差异）；性能基准达标（≤ 明确的资源上限）。

| 阶段 | 动作 | 风险 | 优先级 |
|------|------|:---:|:---:|
| **Phase 3b** | DuckDB + Spark CDP builder 实现 + 属性测试 + 性能基准 | 高——引擎内 SHA256/ROUND/CAST 行为差异 | **P0** |

#### Phase 3c —— Shadow 双跑验证（P0，依赖 3b）

**目标**：在固定回归集上影子模式运行 CDP 摘要路径，与 legacy 路径并行——CDP 路径产生 `DigestExecutionEnvelope` 但不影响最终结论。

**交付物**：
- 摘要路径挂载到 `verify()` 管线，以 shadow 模式与 legacy `_compute_diffs()` 并行
- 固定回归集（≥50 个真实模板）上的双跑报告：CDP digest vs legacy result 对比
- 差异案例分析：每个 CDP≠legacy 的案例必须人工确认根因

**门禁**：Shadow 双跑报告通过 §11 准入标准 #1-7：
- 黄金向量双引擎一致（§11.1 #1）
- 重复执行 digest 稳定（§11.1 #2）
- 已知差异 100% 检出（§11.1 #3）
- 不支持类型不误判（§11.1 #4）
- spec/snapshot 不一致正确路由（§11.1 #5）
- 大结果集 digest 覆盖全量（§11.1 #6）
- Shadow 零假一致（§11.1 #7）

| 阶段 | 动作 | 风险 | 优先级 |
|------|------|:---:|:---:|
| **Phase 3c** | Shadow 双跑：CDP 摘要路径 + legacy 路径并行，验证 count/digest 稳定性和差异检出 | 中——可能发现引擎差异需要回退 3b | **P0** |

#### Phase 3d —— 摘要路径接管结论（P0，依赖 3c 全部门禁通过）

**前置条件**：§11 准入标准 #1-8 **全部通过**。

**动作**：
- `verify()` 结论切换为 CDP 摘要判定：`CONSISTENT_SAMPLE` / `DIFFERENT` / `NOT_EXECUTED` / `UNSUPPORTED_SEMANTICS` / `HUMAN_REVIEW`

| 阶段 | 动作 | 风险 | 优先级 |

## 10. 残余风险

| # | 风险 | 等级 | 说明 |
|---|------|:----:|------|
| R1 | **SQL 方言兼容性**：DuckDB `STRING_AGG` vs Spark `collect_list` 的排序行为是否完全一致 | **中** | 需要双引擎真实数据验证。多重集分桶策略已规避部分风险 |
| R2 | **ROW 类型嵌套**：如果结果包含嵌套类型，CDP 编码可能产生不同结果 | 低 | 当前输出列主要是标量类型；v1 对嵌套类型返回 UNSUPPORTED_SEMANTICS |
| R3 | **浮点 ROUND 精度**：`ROUND(x, N)` 在 SQL 和 PySpark 中的边界值（x.5 恰在中间）、负零行为、NaN/Inf token | **中** | 即使两侧都使用 IEEE 754，`ROUND_HALF_UP` 实现可能不同——必须用黄金向量逐列确认 |
| R4 | **大结果集分桶性能**：>100K 行时多重集分桶的内存和耗时尚无基准数据 | **中** | 需性能基准测试。samples 行数上限已提供输出保护 |
| R5 | **PK samples 不足以诊断**：确定性样本可能无法准确定位所有差异行 | 低 | samples 目的是"快速定位差异模式"而非完整诊断。如需完整差异列表，启动 bucket drill-down |
| R6 | **ORDER BY 在双引擎中的排序稳定性**：非唯一主键的相同值但不同行排序可能不一致 | **中** | 当前分层样本已使用 `ORDER BY CDP_pk_bytes, row_hash`——row_hash 作为稳定次级排序键，不要求唯一 PK |
| R7 | **CDP v1 类型覆盖不完整**：未来出现新的列类型需扩展 CDP | 低 | CDP 版本化——新增类型走新版本，不影响已有行为 |
| R8 | **VARCHAR 无 Unicode 归一化**：相同语义的不同 Unicode 编码（NFC/NFD）会被判定为不同 | 低 | 双引擎 VARCHAR 值通常来自同一上游，编码一致 |
| R9 | **EnvironmentManifest 默认 UNKNOWN**：NaN/Inf 在无 manifest 时回退 HUMAN_REVIEW | **中** | 保守安全——未来正式接入后需业务方声明策略 |
| R10 | **生产环境中 grouping_keys 可能为空**：某些 step 类型不产生 grouping_keys | 低 | 多重集 digest 不依赖 PK；samples 使用 CDP row_hash 排序取前 N 行，诊断能力降低 |

### 12.3 状态语义

| 状态 | 含义 | 判定来源 |
|------|------|---------|
| `CONSISTENT_SAMPLE` | 全量 count + 全量 digest 一致（限定于同一不可变快照，不代表业务正确或生产批准） | 前提条件链全部满足 + count_match=True + digest_match=True |
| `DIFFERENT` | count 或 digest 不一致 | count_match=False 或 digest_match=False |
| `NOT_EXECUTED` | 任一引擎未成功执行 | 引擎执行失败、超时 |
| `UNSUPPORTED_SEMANTICS` | CDP 不支持该类型/场景 | CDP 版本无法覆盖的类型族 |
| `HUMAN_REVIEW` | 系统无法确定结论 | CDP 配置不完整、类型未知、异常状态 |

### 12.4 方案适用边界

**覆盖范围内**（可自动判定 `CONSISTENT_SAMPLE` 或 `DIFFERENT`）：
- 关系型、标量列、同一不可变快照
- 两侧输出列集合、列顺序和类型族已确定
- 两侧使用同一 CDP 版本和同一 `CreDigestSpec`
- 所有输出类型都在 CDP 支持范围内
- 执行成功且摘要确实覆盖全量结果

**不覆盖**（自动进入 `UNSUPPORTED_SEMANTICS`、`NOT_EXECUTED` 或 `HUMAN_REVIEW`）：
- CDP 不支持的 STRUCT/LIST/MAP、二进制或引擎特有类型
- 缺失列类型、时区或浮点规范，无法生成相同 canonical bytes
- 任一引擎未执行、执行失败、只计算了部分数据
- 摘要协议版本或 schema digest 不一致
- 需要定位具体差异单元格（需 bucket drill-down）
