# Pipeline 主链路详解——从 DeveloperSpec 到 ReviewPackage

> 基于 `src/tianshu_datadev/api/pipeline.py` 的 `run_all()` 方法实际代码路径追踪。
>
> 相关文档：[[规划合理性评估与技术架构分析_20260629_2115]]

---

## 1. 全链路一览

```
DeveloperSpec Markdown
    │
    ▼
[Stage 1] DeveloperSpecParser ─── 确定性解析 → ParsedDeveloperSpec + SourceManifest
    │
    ▼
[Stage 2] SpecEnricher ────────── 推断缺失指标 → 补充 window_metrics / computed_metrics
    │
    ├─ RelationshipPlanner ─────── 多表 Join 推断 → RelationshipHypothesis（四档证据等级）
    ├─ cross_validate ──────────── 指标推断 ↔ Join 推断一致性检查
    │
    ▼
[Stage 3] SqlBuildPlanBuilder ──── spec + hypothesis → SqlBuildPlan（8 step 类型化 IR）
    │                              └→ SqlProgramBuilder → SqlProgram（多语句 DAG 容器）
    ▼
[Stage 3b] SqlBuildPlanValidator ── Plan 结构合法性 + 多跳链循环/深度检测
    │
    ▼
[Stage 4] DuckDbSqlCompiler ────── SqlBuildPlan → 确定性渲染 SQL 字符串
    │
    ▼
[Stage 5] DuckDBExecutor ───────── 执行 SQL → ExecutionTrace + ResultSummary
    │
    ▼
[Stage 6] DataTransformContractExtractor ── 从已验证 SqlProgram 抽取引擎无关契约
    │
    ▼
[Stage 7] ReviewPackageBuilder ──── 打包全部产物 → ReviewPackage（供人工审查）
```

---

## 2. 各 Stage 详解

### Stage 1：DeveloperSpecParser（确定性解析）

**文件**：`developer_spec/parser.py` + `developer_spec/source_manifest.py`

**输入**：DeveloperSpec Markdown 文本（程序员编写的半结构化声明）

**输出**：
- `ParsedDeveloperSpec`：结构化 spec，含 `input_tables`、`metrics`、`dimensions`、`joins`、`compute_steps`、`output_spec` 等
- `SourceManifest`：表字段事实源追踪——每个字段标记来源（`developer_spec` / `schema_registry`）

**确定性保障**：
- `spec_hash`（SHA-256 归一化哈希）：相同输入 → 相同 hash，用于幂等性和缓存键
- 6 项允许宽松 + 7 项禁止宽松——纯自然语言输入直接 REJECT
- `SOURCE_CONFLICT` 检测：DeveloperSpec 声明与 SchemaRegistry 冲突时不静默覆盖

**关键调用链**：
```python
parser = DeveloperSpecParser()
spec = parser.parse(markdown_text)          # Markdown → ParsedDeveloperSpec
manifest = build_manifest_from_spec(spec)   # spec → SourceManifest
```

---

### Stage 2：SpecEnricher + RelationshipPlanner（推断与补充）

**文件**：`planning/spec_enricher.py` + `planning/relationship_planner.py` + `planning/cross_validator.py`

**三个子步骤**：

#### 2a. SpecEnricher（指标推断）

**输入**：`ParsedDeveloperSpec` + `SourceManifest`

**推断内容**：
- **窗口指标**（`inferred_window_metrics`）：从业务描述中识别"环比"、"同比"、"排名"等，推断需要 LAG / RANK / SUM_OVER 等窗口函数
- **计算指标**（`inferred_computed_metrics`）：识别"客单价 = 交易额 / 订单数"等派生指标，将 `expression` 字段填充为安全的列引用表达式
- **标签规则**（`label_rules`）：从描述中推断 CASE WHEN 分支逻辑

**双重实现**：
- `FakeSpecEnricher`：正则匹配 + 规则推断，Phase 4 前使用
- `SpecEnricher`：LLM 推断，注入 `ProviderAdapter` 后启用；adapter=None 时退化为 Fake

**输出**：补充了 `inferred_window_metrics`、`inferred_computed_metrics`、`label_rules` 的 `ParsedDeveloperSpec`

#### 2b. RelationshipPlanner（Join 推断）

**输入**：`ParsedDeveloperSpec` + `SourceManifest`

**行为**：
- **单表**：跳过，`hypothesis = None`
- **多表**：推断表间 Join 关系 → `RelationshipHypothesis`

**双重实现**：
- `FakeRelationshipPlanner`：仅处理 DeveloperSpec 中显式声明的 `JoinDecl`，调用 `RelationshipValidator` 确定性定级（STRONG/MEDIUM/WEAK/NONE），WEAK/NONE 硬门禁
- `RelationshipPlanner`（Phase 4E）：adapter 已注入时调用 LLM 推断隐式 Join + 合并显式声明 + Validator 定级

**输出**：`RelationshipHypothesis`——含 `candidates`（JoinCandidate 列表，每个带 `evidence` 证据链 YAML）

#### 2c. cross_validate（交叉验证）

**输入**：`spec` + `hypothesis` + `manifest`

**行为**：检查 SpecEnricher 推断的指标是否与 RelationshipPlanner 推断的 Join 关系一致——指标引用的字段是否在 Join 可达的表中存在。

**关键调用链**：
```python
spec = self._spec_enricher.apply_enrichment(spec, manifest)
if len(spec.input_tables) > 1:
    hypothesis, extra_questions = self._relationship_planner.plan(spec, manifest)
if hypothesis:
    xv_questions = cross_validate(spec, hypothesis, manifest)
```

---

### Stage 3：SqlBuildPlanBuilder + SqlProgramBuilder（IR 构建）

**文件**：`planning/sql_build_plan.py` + `planning/sql_program.py`

**输入**：`ParsedDeveloperSpec` + `RelationshipHypothesis`

**三条构建路径**：

| 路径 | 触发条件 | 方法 | 产物 |
|------|---------|------|------|
| **单表/简单两表** | 无 `compute_steps` + 候选 ≤ 1 | `builder.build(spec, hypothesis)` | 单个 `SqlBuildPlan` → `build_sql_program()` → STANDALONE SqlProgram |
| **多跳 Join 链** | 无 `compute_steps` + 候选 > 1 | `builder.build_multi(spec, hypothesis)` | Plan 链 → `build_sql_program_from_chain()` → PRODUCER→…→FINAL SqlProgram |
| **ComputeSteps DAG** | `compute_steps` 非空 | `builder.build_from_steps(spec, hypothesis)` | Plan DAG → `build_sql_program_from_compute_steps()` → 含分支/合流的 SqlProgram |

**SqlBuildPlan 的 9 Step 类型化 IR**（所有 step 均为 `StrictModel(extra="forbid")`，禁止 `raw_sql`）：

```
ScanStep → FilterStep → JoinStep → AggregateStep → WindowStep
→ CaseWhenStep → ProjectStep → SortStep → LimitStep
（+ SubqueryStep，Phase 4.6 Step 2 新增）
```

**SqlProgramBuilder 职责**（详见 [[SqlProgramBuilder 边界收紧]]）：
- 消费 `SqlBuildPlanBuilder` 产出的 Plan 列表
- 按显式依赖（plan 顺序 / `compute_steps.source`）推导 `StatementKind` + `depends_on` + `produces`
- 确定性组装 `SqlProgram`（含 `_temp` 中间表 + 拓扑排序 + 生命周期校验）

**关键调用链**：
```python
builder = SqlBuildPlanBuilder()
if spec.compute_steps:
    plans = builder.build_from_steps(spec, hypothesis)
    sql_program = build_sql_program_from_compute_steps(plans, spec, chain_id)
elif len(hypothesis.candidates) > 1:
    plans = builder.build_multi(spec, hypothesis)
    sql_program = build_sql_program_from_chain(plans, spec.spec_hash, chain_id)
else:
    plan, _ = builder.build(spec, hypothesis)
    sql_program = build_sql_program(plan, spec.spec_hash)
```

---

### Stage 3b：SqlBuildPlanValidator（结构校验）

**文件**：`sql/validator.py`

**校验维度**：
- **单 Plan**：`validate(plan, manifest)`——Scan table_ref 存在性、Join key 列存在性、ColumnRef 完整性、窗口函数白名单
- **多跳链**：`validate_multi_hop_chain(sql_program)`——V-009b 循环检测（右表引用链中不得重复）、V-009c 深度上限（JoinStep 总数 ≤ 5）
- **DAG**：`validate_program_dag(sql_program)`——循环检测、缺失依赖、消费者授权、生产者可达性

---

### Stage 4：DuckDbSqlCompiler（确定性编译）

**文件**：`sql/compiler.py`

**输入**：`SqlProgram`（含 `SqlBuildPlan` 列表）

**编译策略**：
- **单语句**：`compile(plan)` → `SqlArtifact`（含 SQL 字符串 + 参数列表）
- **多语句**：`compile_program(sql_program)` → `ProgramCompiledSql`（每个语句的 SQL + CREATE TEMP TABLE + cleanup DROP）

**确定性保障**：
- 相同 `SqlBuildPlan` 两次编译 → 字节一致 SQL + 同一 SHA-256 hash
- Compiler 不拼接任何自由 SQL——所有 SQL 片段来自 Step IR 的结构化渲染
- 4 个优化 Pass：谓词下推、列裁剪、常量折叠、Join 重排序

**编译产物结构**（多语句）：
```python
ProgramCompiledSql(
    statements=[
        CompiledStatement(statement_id="stmt_agg",
                          sql="CREATE TEMP TABLE _temp_cX_Y AS SELECT ..."),
        CompiledStatement(statement_id="stmt_output",
                          sql="SELECT ... FROM _temp_cX_Y ..."),
    ],
    cleanup_sql=["DROP TABLE IF EXISTS _temp_cX_Y"],
    statement_order=["stmt_agg", "stmt_output"],  # 拓扑排序的执行顺序
)
```

---

### Stage 5：DuckDBExecutor（执行）

**文件**：`sql/executor.py`

**输入**：`ProgramCompiledSql`（来自 Compiler）

**执行流程**：
1. 按 `statement_order` 逐一执行 SQL 语句
2. 每条语句执行后用 `ResultSummary` 记录行数、列数、耗时、NULL 统计
3. 中间语句失败 → 立即阻断后续语句 → cleanup SQL 仍然执行（`_temp` 表清理不因失败跳过）
4. 全部成功 → cleanup → 返回 `ProgramExecutionResult`

**输出**：
```python
ProgramExecutionResult(
    results=[StatementResult(statement_id, trace, summary), ...],
    completed_count=2,
    failed_at=None,          # 失败时标记失败的 statement_id
    cleanup_status="success"  # cleanup 独立执行，不受语句失败影响
)
```

---

### Stage 6：DataTransformContractExtractor（契约抽取）

**文件**：`artifacts/contract_extractor.py`

**输入**：已验证并执行成功的 `SqlProgram`

**抽取逻辑**（纯确定性——不依赖 LLM）：
- 从 `SqlBuildPlan` 的 steps 中抽取引擎无关的结构化规格
- `ScanStep` → `declared_inputs` + `field_schemas`
- `JoinStep` → `join_paths`（含 left_table / right_table / keys / type）
- `AggregateStep` → `metrics`（含 aggregation / input_column / alias / filter）
- `ProjectStep` → `output_schema`（含 name / type / nullable）
- `WindowStep` → `window_specs`

**输出**：`DataTransformContract`——Spark 侧的权威输入契约

**关键设计**（详见架构分析文档 §1.1）：
- Spark 侧不重新读 DeveloperSpec，不重新理解业务需求
- DataTransformContract 从 SQL 侧已验证的 SqlBuildPlan 确定性抽取——Spark 侧只做引擎翻译
- 交叉验证时差异 = 翻译问题（可精确定位到 step），而非 "两个 LLM 理解不一致"（排查困难）

---

### Stage 7：ReviewPackageBuilder（审查包打包）

**文件**：`artifacts/packager.py`

**输入**：`PackageInputs`——包含全部上游产物的引用

**打包内容**：
- 原始 DeveloperSpec Markdown
- `ParsedDeveloperSpec`（结构化 spec）
- `SqlBuildPlan`（step IR）
- 编译后的 SQL 文本
- 执行结果（`ExecutionTrace` + `ResultSummary`）
- `DataTransformContract`
- `OpenQuestion` 列表（含 Join 证据链 YAML、人审待确认项）
- 审查包清单（SHA-256 完整性校验）

**输出**：`ReviewPackage`——供程序员审查的完整数据包

## 3. 三条构建路径的决策树

```
                     DeveloperSpec Markdown
                            │
                     [Stage 1] Parser
                            │
                     [Stage 2] Enrich + Plan
                            │
                    spec.compute_steps 非空？
                         /        \
                       是          否
                       /            \
               build_from_steps    hypothesis.candidates > 1?
                    │                 /        \
                    │               是          否
                    │               /            \
                    │       build_multi      build()
                    │            │               │
                    │      多 Plan 线性链     单 Plan
                    │            │               │
                    └────────────┴───────────────┘
                                 │
                    SqlProgramBuilder（消费显式依赖，确定性组装）
                                 │
                          SqlProgram
                                 │
                     [Stage 3b] Validator
                                 │
                     [Stage 4] Compiler
                                 │
                     [Stage 5] Executor
                                 │
                     [Stage 6] Contract Extractor
                                 │
                     [Stage 7] Package Builder
                                 │
                           ReviewPackage
```

## 4. 关键设计原则

### 4.1 每阶段失败独立捕获

Pipeline 采用 "try/except per stage" 模式——任一阶段失败，已完成的前序产物保留至 `self._results`，返回含 `pipeline_error` + `pipeline_stages` 的部分结果。不存在"全有或全无"的失败模式。

### 4.2 LLM 与确定性代码的边界

| 确定性模块（不能被 LLM 覆盖） | LLM 角色（辅助加速） |
|---|---|
| DeveloperSpecParser——确定性解析 | SpecEnricher——推断缺失指标 |
| SourceManifest——事实追踪 | RelationshipPlanner——提 Join 候选 |
| SqlBuildPlanBuilder——确定性构建 IR | SparkDeveloper——生成代码草案 |
| SqlBuildPlanValidator——结构校验 | |
| DuckDbSqlCompiler——确定性渲染 SQL | |
| DuckDBExecutor——执行 + 结果采集 | |
| DataTransformContractExtractor——确定性抽取契约 | |
| ReviewPackageBuilder——确定性打包 | |

### 4.3 Fake Adapter 策略

Phase 1-4 全部开发工作不依赖真实 LLM——`SpecEnricher` 和 `RelationshipPlanner` 均提供 Fake 实现（纯规则/正则推断）。Phase 4 注入 `ProviderAdapter` 后无缝切换到 LLM 模式。这保证了早期开发的可测试性和确定性。
